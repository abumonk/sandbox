## Description
Extend `get_adventure()` to include a `summary` object with TC counts, task-status counts, and `next_action` (see design-backend-endpoints). Add `compute_next_action()` helper. Stdlib-only. No new fields persisted.

## Acceptance Criteria
- [ ] `GET /api/adventures/{id}` responses include a `summary` block with all declared fields (`tc_total`, `tc_passed`, `tc_failed`, `tc_pending`, `tasks_by_status`, `next_action`).
- [ ] `compute_next_action` handles every state enum value in `VALID_STATES` (`concept`, `planning`, `review`, `active`, `blocked`, `completed`, `cancelled`) plus an `unknown` fallback, and returns one of the five allowed `kind` values.
- [ ] When state is `review` and `permissions.status` is not `"approved"`, `next_action.kind == "approve_permissions"` (TC-029).
- [ ] No new third-party imports added — `server.py` top-level imports remain a subset of the stdlib allowlist (TC-030).

### Summary
Two server-side additions, both stdlib-only, both in `.agent/adventure-console/server.py`:

1. A new pure helper `compute_next_action(meta, permissions, tcs, tasks)` that maps the adventure `state` (plus `permissions.status`) to a `{kind, label, state_hint}` dict. Kinds: `approve_permissions`, `open_plan`, `resolve_blocker`, `promote_concept`, `none`.
2. An extension of `get_adventure()` that bundles TC-status counts, task-status counts, and the `next_action` result under a new top-level `summary` key.

### Implementation Steps
1. **Insert bucketing helpers** after `_target_conditions()` (~line 178): `_TC_PASSED_TOKENS`, `_TC_FAILED_TOKENS`, and `_bucket_tc_status(raw)` that normalizes raw TC table `status` cells into `passed` / `failed` / `pending`.
2. **Insert `compute_next_action()`** as a module-level function (so tests can import it). State dispatch: `blocked → resolve_blocker`, `concept → promote_concept`, `review + unapproved → approve_permissions`, `review + approved → open_plan`, `planning → open_plan`, `active → open_plan` (label "Continue Execution"), everything else → `none`. Always echo the lower-cased state as `state_hint`.
3. **Extend `get_adventure()`**: just before the `return {...}` dict (~line 262), compute `tc_buckets` from `tcs`, `tasks_by_status` from `tasks`, and `summary = { tc_total, tc_passed, tc_failed, tc_pending, tasks_by_status, next_action: compute_next_action(meta, permissions, tcs, tasks) }`. Add `"summary": summary,` to the returned dict.
4. **Do not touch** `list_adventures()`, any mutation handler, or the HTTP routing. Do not add any `import` statements.
5. **Tests** go in `.agent/adventures/ADV-009/tests/test_server.py` (created under ADV009-T001's test strategy). Required cases:
   - `test_summary_block_shape` — asserts the six required keys exist on a real adventure fixture (TC-026).
   - `test_next_action_review_unapproved_permissions` — TC-029.
   - `test_next_action_handles_every_valid_state` — iterates `VALID_STATES` + `unknown`; verifies allowed `kind` enum and `state_hint` echo.
   - `test_stdlib_only_imports` — parses `server.py` and asserts the import set is within the stdlib allowlist (TC-030).
6. **Verify** by running `python -m unittest discover -s .agent/adventures/ADV-009/tests -p "test_*.py"` and by `curl`ing `GET /api/adventures/ADV-009` against a running server and pretty-printing the `summary` block.

### Target Files
- `.agent/adventure-console/server.py` — add `_bucket_tc_status`, `compute_next_action`, extend `get_adventure()` return dict with `summary`.
- `.agent/adventures/ADV-009/tests/test_server.py` — add the four test cases above (file itself is created under the umbrella of ADV009-T001; this task just appends its four functions).

### Risks
- Renaming `next_action.kind` later will silently break the Overview tab; the shape test locks it.
- TC status vocabulary is free-form text from the manifest table; unknown tokens default to `pending`, which is the safe/conservative bucket.
- Summary is recomputed per request — acceptable given the small per-adventure footprint.
