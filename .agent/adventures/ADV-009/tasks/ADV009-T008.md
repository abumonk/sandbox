## Description
Group tasks by `status` into buckets (hide empty ones). Render each task as the new card (status pill + ID + title + depends-on + TC list). Build the expanded detail panel with status strip, depends-on chain, TC checklist, description, acceptance criteria, and "Show details" disclosure for frontmatter/log/raw. Parse Description and Acceptance Criteria sections client-side from the task file.

## Acceptance Criteria
- [ ] Tasks group by status bucket; empty buckets hidden.
- [ ] Task card hides file path, assignee, iterations by default.
- [ ] Detail panel renders Description and Acceptance Criteria as structured components (not a markdown dump).
- [ ] Show-details disclosure contains frontmatter + log + raw source.
- [ ] TC checklist reflects each TC's current status.

## Design
See `.agent/adventures/ADV-009/designs/design-task-card-layout.md` for the
full component spec. Concrete implementation steps for this task:

### Scope

Replace the flat sortable table in `renderTasks` and the markdown-dump in
`openTaskDetail` with (a) a status-bucketed card list and (b) a structured
detail panel, both defined as new functions alongside the existing ones in
`.agent/adventure-console/index.html`. Touch only that one file. Keep v1
renderers untouched — T011 retires them after T007/T008/T009/T010 land.

### Implementation Steps

1. **Add a section-parser helper** (top of the `App = {...}` object, near
   other helpers): `parseTaskSections(src)` that splits frontmatter (`---`
   fences) from body, then uses a `/^## (Description|Acceptance Criteria|Log)\s*$/m`
   regex to slice the body into `{frontmatter, description, acceptance,
   log, raw}`. Frontmatter is returned as the raw YAML text (no parse —
   the task list from `/api/adventures/{id}` already provides typed
   fields). Acceptance criteria returned as an array of line strings with
   leading `- [ ]` / `- [x]` preserved so the UI can render checkboxes.

2. **Replace `renderTasks(a)`** with a bucketed layout:
   - Bucket order: `pending`, `in_progress`, `passed`, `failed`, `done`.
     Unknown statuses fall into an "Other" bucket at the end.
   - For each non-empty bucket render `<section class="stack">` with an
     `<h3>` header `{Bucket} (N)` and a column of task cards.
   - Cards use the new `.card` primitive (already in page from T005) and
     are created by a new `renderTaskCard(t, a)` helper. Card content, in
     order:
     1. Top row: status `.pill.status-{t.status}` + monospace `t.id` +
        muted `{stage}`.
     2. Title line (`.card-title` class from visual system).
     3. Meta line: `depends_on: T001 · T003` (dots between IDs; omit the
        whole line if the list is empty) and, right-aligned, `TCs: TC-008,
        TC-009` (omit if empty).
   - Click handler on the whole card calls `openTaskDetail(t, a)` (same
     inline-expand target `#task-detail` as today — keep that id).
   - `cursor: pointer`, `data-testid="task-card-${t.id}"` for the test
     hooks called out in design-test-strategy.
   - Do **not** render `t.file`, `t.assignee`, or `t.iterations` on the
     card (TC-009).

3. **Replace `openTaskDetail(t)` → `openTaskDetail(t, a)`** and build the
   detail panel from structured DOM, not `marked.parse(txt)`:
   1. Fetch raw task file via `API.file(t.file)`; pass text through
      `parseTaskSections`.
   2. **Status strip**: a `.card` containing status pill, stage chip,
      assignee badge, and `iterations: N` — these are the fields the card
      intentionally hides, surfaced on expand.
   3. **Depends-on chain**: inline nodes built from `t.depends_on`. Each
      predecessor resolved against `a.tasks` to color a status dot
      (`.status-dot.status-{status}`). Render `T001 → T003 → (this)` with
      `→` separators. Omit the whole card if `depends_on` is empty.
   4. **TC checklist**: for each id in `t.target_conditions`, look up the
      TC in `a.target_conditions` (already on the adventure bundle — see
      line 462 of the current file). Render a `<label class="tc-row">`
      with an unchecked `<input type="checkbox" disabled>` for
      `status !== 'passed'` and checked for `status === 'passed'`, then
      the TC id and description. A failed TC gets `.status-failed` text
      coloring. Omit the card if the list is empty.
   5. **Description**: wrap parsed description text in a `.card` and
      render with `marked.parse` so inline markdown (code, links) still
      works — this is prose from one section, not the whole file, so it
      meets TC-010.
   6. **Acceptance Criteria**: iterate the parsed array. Each `- [ ]` /
      `- [x]` line becomes `<li><input type="checkbox" disabled
      {checked?}> {rest of line}</li>`. Non-checkbox lines render as
      plain `<li>`. Wrap in a `.card` with title "Acceptance Criteria".
   7. **Show details** disclosure: a single `<details class="disclosure">`
      whose summary reads `Show details` and whose body contains, in
      order: a `<pre>` of the frontmatter YAML, a muted line `File:
      {t.file}`, a `<pre class="log-tail">` of the `## Log` section
      (using existing `.log-tail` class), and a `<pre>` of the full raw
      source. This satisfies TC-011.

4. **Scroll + active-card polish**: when a card is expanded, set
   `.active` class on that card and clear it from siblings (mirrors the
   existing `.file-entry.active` pattern). Scroll detail panel into view
   via `detail.scrollIntoView({block:'nearest'})`.

5. **Empty-state handling**: if `a.tasks.length === 0`, keep the current
   `'No tasks.'` empty message. If every bucket ends up empty after
   grouping (shouldn't happen given the above, but guard anyway), fall
   through to the same empty message.

6. **Testids** (for T012 browser tests per design-test-strategy): add
   `data-testid="task-bucket-{status}"` on each bucket section,
   `data-testid="task-card-{id}"` on each card,
   `data-testid="task-detail"` on the expanded panel container,
   `data-testid="tc-row-{tc_id}"` on each TC checklist row,
   `data-testid="task-show-details"` on the disclosure summary.

7. **Do NOT touch in this task**:
   - `renderFileBrowser`, `renderPermissions`, `renderKnowledge`,
     `renderLog`, `renderReviews` — T011 removes those.
   - The tab dispatch in `switchTab` — T006 already routes `tasks` to
     `renderTasks`.
   - Server endpoints — no backend changes needed; the adventure bundle
     already carries `tasks[].target_conditions`, `tasks[].depends_on`,
     `tasks[].file`, `tasks[].status`, `tasks[].stage`, and
     `a.target_conditions[]` with `status` (confirmed at current line
     462 of index.html).

### Risks / Notes
- `marked.js` is still loaded (for description prose and Documents tab
  later). That's intentional and not a regression.
- The `## Log` section in a task file may contain lines that look like
  markdown lists; we render it inside `<pre>` so prefixing `- ` bullets
  render literally. That matches v1 visual intent.
- The parser must tolerate tasks that don't yet have a `## Description`
  or `## Acceptance Criteria` section (older tasks). In that case the
  corresponding card is omitted rather than rendering an empty
  placeholder.
- No new CSS is introduced here — all styling reuses `.card`, `.pill`,
  `.stack`, `.disclosure`, `.chip`, `.log-tail` from T005. If any helper
  class (e.g., `.tc-row`, `.status-dot`, `.card-title` if not in T005)
  turns out to be missing, add a minimal rule inline at the top of the
  existing `<style>` block and call it out in the PR description so
  T005's CSS can absorb it in a follow-up.
