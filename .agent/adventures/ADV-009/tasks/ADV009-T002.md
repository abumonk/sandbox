## Description
Produce `.agent/adventures/ADV-009/research/audit.md` cataloging every UI element in the current console with a `keep / hide-behind-toggle / remove` verdict and a v2 target surface. Drive the verdicts from the user's requirements in the manifest concept (5-second glance, no raw paths / frontmatter / log dumps by default).

## Acceptance Criteria
- [ ] >= 30 distinct elements enumerated.
- [ ] Every row has a `verdict` in the allowed set.
- [ ] Every current-tab dispatch branch (overview/designs/plans/tasks/permissions/reviews/knowledge/research/log) has >= 1 row.
- [ ] Summary line at top: "X kept * Y hidden * Z removed".

## Design

Authoritative design reference: `.agent/adventures/ADV-009/designs/design-simplification-audit.md`.
Supporting v2 target surfaces come from `.agent/adventures/ADV-009/designs/design-information-architecture.md` (four-tab shape: Overview, Tasks, Documents, Decisions).

### Approach
Catalog every UI element currently rendered by `.agent/adventure-console/index.html` and classify each as `keep`, `hide-behind-toggle`, or `remove`. The verdicts are driven by the manifest concept's 5-second-glance requirement — anything that is raw path, frontmatter, log dump, or command-line noise defaults to `hide-behind-toggle` or `remove` unless it answers a question within 5 seconds of opening the adventure.

### Target Files
- `.agent/adventures/ADV-009/research/audit.md` (new) — the catalog table.
- Read-only inputs: `.agent/adventure-console/index.html`, `.agent/adventure-console/server.py`, `.agent/adventures/ADV-009/manifest.md`.

### Implementation Steps
1. Read `index.html` end-to-end and enumerate every visible element produced by each render function, keyed to the `renderTab` dispatch in the file:
   - Sidebar (`refreshList`): Claudovka/team-pipeline logo, refresh button, adventure row (id, state badge, title).
   - Header (`render`): ADV id, title, state pill, created/updated/tasks meta line, state-transition button row.
   - Tabs bar: 9 tabs — overview, designs, plans, tasks, permissions, reviews, knowledge, research, log (plus the per-tab count suffix).
   - Overview pane (`renderOverview`): Concept box, Target Conditions table (ID, Description, Task(s), Proof command, Status), Adventure Report shortcut button.
   - Tasks pane (`renderTasks`): task table (ID, Title, Stage, Status, Assignee, Iter, TCs) + per-task detail drawer rendered from raw markdown.
   - Designs/Plans/Research panes (`renderFileBrowser`): file list + rendered file view + Approve button (designs only).
   - Permissions pane (`renderPermissions`): status badge, approved-by line, Approve button, inline markdown view.
   - Reviews pane (`renderReviews`): adventure-report.md star row, per-task review rows (task_id + status badge), file view.
   - Knowledge pane (`renderKnowledge`): suggestions list with checkboxes, Record-Selection / Select-All / None buttons.
   - Log pane (`renderLog`): "Adventure Log (tail)" header + raw log tail `<pre>`.
   - Global chrome: toast banner, raw paths under file entries, file-browser split-pane.
2. For each element record one row in a single markdown table with columns: `Element | Current tab | Verdict | Reason | New home (v2)`.
   - `Verdict` is from the allowed set `{keep, hide-behind-toggle, remove}`.
   - `New home (v2)` is one of `Overview | Tasks | Documents | Decisions | Show-details drawer | (removed)` per design-information-architecture.md.
3. Apply these defaults when assigning verdicts:
   - `keep` — the element answers a 5-second-glance question (state pill, TC progress, task grouping, concept synopsis, next-action card).
   - `hide-behind-toggle` — useful for debugging or deep reading but visually noisy (raw frontmatter, log tail, command column, raw file paths, iteration counts, file-list counts).
   - `remove` — superseded by v2 aggregation (standalone Log/Knowledge/Permissions/Designs/Plans/Research/Reviews tabs; duplicate buttons; unused state-transition shortcuts).
4. Ensure ≥ 1 row per `renderTab` branch (overview, designs, plans, tasks, permissions, reviews, knowledge, research, log). Target total ≥ 30 rows; the enumeration above yields ~40 rows, so leave headroom.
5. At the very top of `audit.md`, write the summary line in the exact form: `X kept * Y hidden * Z removed` (ASCII asterisks, not middots) where X+Y+Z equals the row count.
6. Add a short tail section "Open Questions" listing any element whose verdict required a judgment call (e.g., state-transition buttons moving to Decisions vs. staying in header) so downstream frontend tasks can reconcile.

### Row Template
```markdown
| Element | Current tab | Verdict | Reason | New home (v2) |
|---------|-------------|---------|--------|---------------|
| Log tail pane | log | hide-behind-toggle | Debug-only; violates 5s-glance when shown by default | Show-details drawer (Overview) |
```

### Testing Strategy
Manual verification against acceptance criteria:
- Count rows, confirm ≥ 30.
- Grep `| (overview|designs|plans|tasks|permissions|reviews|knowledge|research|log) |` to prove every dispatch branch has ≥ 1 row.
- Confirm every row's `Verdict` column is in `{keep, hide-behind-toggle, remove}`.
- Confirm top-of-file summary matches the row-count totals.

### Risks
- Duplicate-element risk: the same concept (e.g., "file view") exists in three panes. Enumerate per pane to hit coverage rather than dedupe across panes.
- Verdict drift: if design-information-architecture.md is later revised, this audit must be treated as the snapshot of intent at T002 time and revisited rather than silently edited.
- Summary-line format must be exact — downstream tasks may grep for it.
