## Description
Build `renderDecisions` with three conditionally rendered cards: state transitions, pending permissions (with parsed request counts), and knowledge suggestions (reusing `parseKnowledgeSuggestions` and the `/knowledge/apply` POST). Hide cards that have no actionable content. Move the per-design Approve button into `renderDesignDoc` (T009), not here.

## Acceptance Criteria
- [ ] Three cards render when applicable; empty cards hidden.
- [ ] Permissions card shows counts of shell/file/MCP requests parsed client-side; full doc behind disclosure.
- [ ] State-transition buttons post to `/api/adventures/{id}/state`.
- [ ] Knowledge card writes same JSON payload as v1 (regression parity).

## Design

- `.agent/adventure-console/index.html` — add `renderDecisions` method to `App` object and a `parsePermissionCounts` helper function

### Implementation Steps

**Step 1 — Add `parsePermissionCounts(md)` helper (standalone function, near `parseKnowledgeSuggestions`)**

Parse the raw permissions markdown to extract counts by request type. The v1 permissions doc uses `## Requests` with table rows like `| 1 | shell | ... |`. Scan lines for pipe-delimited rows, extract column 3 (the type field: `shell`, `file-read`, `file-write`, `mcp`, etc.), and tally into an object `{ shell: N, file: N, mcp: N }`. Collapse `file-read` and `file-write` into a single `file` bucket. Return the counts object. This is a pure function — no DOM, no side effects.

**Step 2 — Add `renderDecisions(a)` method to the `App` object**

Create the method returning a `div`. It conditionally appends up to three child cards (using the `.card` CSS class from T005). Each card is built by a sub-helper described below. If a card helper returns `null`, skip it. If all three are null, show an empty-state message: "No pending decisions."

Wire into `renderTab()` dispatch: add `if (tab === 'decisions') return pane.appendChild(this.renderDecisions(a));` (T006 will have set up the four-tab bar with `decisions` as the fourth key; if not yet present, the coder adds it).

**Step 3 — Card 1: State Transitions (`renderStateTransitionCard(a)`)**

- Read `STATE_TRANSITIONS[a.state]` (already exists at line ~302).
- If empty array, return `null`.
- Build a `.card` with:
  - Header: `h('h3', {}, 'State Transition')`.
  - Current state shown as a `.pill` with class `status-{state}`.
  - A `.btn-row` with one button per valid next state. Each button calls the existing `this.transitionState(newState)` (line ~736).
  - A `.disclosure` element containing the last 20 log lines (`a.log_tail`), labeled "Recent log" — gives audit context before transitioning.
- `data-testid="decisions-state-card"` on the card root.
- `data-testid="state-transition-btn-{state}"` on each button.

**Step 4 — Card 2: Pending Permissions (`renderPermissionsCard(a)`)**

- If `!a.permissions`, return `null`.
- Build a `.card` with `data-testid="decisions-permissions-card"`.
- If `a.permissions.status !== 'approved'`:
  - Fetch the raw markdown via `API.file(a.permissions.file)`.
  - Call `parsePermissionCounts(md)` to get `{ shell, file, mcp }`.
  - Render a summary strip: "N shell · N file · N MCP requests".
  - Render a `.disclosure` (collapsed by default) with the full permissions markdown rendered through `marked.parse()`.
  - Render an "Approve Permissions" button that calls `API.approvePerm(a.id)` with confirm dialog, then `this.reopen()` — same logic as v1 `renderPermissions`.
- If `a.permissions.status === 'approved'`:
  - Show a neutral line: "Permissions approved on {a.permissions.approved}".
  - A `.disclosure` with the full doc for reference.
- `data-testid="permissions-approve-btn"` on the approve button.
- `data-testid="permissions-summary"` on the summary strip div.

**Step 5 — Card 3: Knowledge Suggestions (`renderKnowledgeCard(pane, a)`)**

- If `!a.adventure_report`, return `null`.
- Build a `.card` with `data-testid="decisions-knowledge-card"`.
- Reuse the existing `parseKnowledgeSuggestions(txt)` function (line ~760) — fetch `API.file(a.adventure_report.file)`, parse, render.
- If `suggestions.length === 0`, return `null` (hide card entirely when no actionable content).
- Render a checkbox list identical to v1's `renderKnowledge`: for each suggestion, a checked `<input type="checkbox">` + title + type + target + body snippet.
- "Record Selection" button calls `API.applyKB(a.id, pickedIndices)` — same payload shape as v1 (array of integer indices). Toast on success/error.
- "Select All" and "None" secondary buttons.
- No raw Section 6 markdown dump, no "Run `/adventure-review`" hint — those move behind a "Show details" `.disclosure` if needed.
- `data-testid="knowledge-record-btn"` on the Record Selection button.
- `data-testid="knowledge-suggestion-{index}"` on each suggestion row.

**Step 6 — Async orchestration**

`renderDecisions` is synchronous for card 1 (state data is in `a` already) but cards 2 and 3 need async fetches. Pattern: render placeholder `<div class="empty">Loading...</div>` inside cards 2 and 3, then replace content when the fetch resolves. This matches the existing v1 pattern used by `renderKnowledge` and `renderPermissions`.
