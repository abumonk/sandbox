# Phase 6.2 — Test & Profile Design

This document specifies the **test and profile design** for the
post-reconstruction Claudovka ecosystem. It covers the full test
taxonomy (unit, integration, E2E, scenario, chaos) that lives above
the per-entity autotests from TC-019, the workload profiles (small,
medium, large) that feed both the functional tests and the benchmarks,
and the coverage mapping from test scenarios to the coverage targets
declared in the autotest strategy.

Where TC-019 defined *which kinds of tests each entity must carry*,
this document defines *which scenarios cross multiple entities* —
the stack traversals that exercise producer/consumer pairs, edge
contracts, full adventure runs, and fault-injection. Think of TC-019
as the unit-test policy and this document as the integration/E2E
policy.

---

## 1. Principles

### 1.1 Pyramid with a broad middle

The conventional pyramid (many unit, few integration, fewer E2E) is
inverted for Claudovka's specific failure modes: the historical
incidents in phase1 (X1-X11) are predominantly cross-entity race
conditions, contract mismatches, and prompt-to-tool drift. These fail
at the *seams*, not within single modules. The design therefore
allocates **more test weight to the integration layer** than a
conventional web app would:

```
               [  chaos  ]            2 %
              [  E2E/scenario ]       8 %
            [  integration  ]         40 %
          [  contract  ]              20 %
        [  unit (per entity)  ]       30 %
```

(Percentages are of test *count* under `scope=full`, not runtime.)

### 1.2 Every scenario names its contracts

An integration or E2E test references at least one edge contract
(E1-E14) by id in its name or docstring. The reviewer gate includes
"does this scenario exercise a declared contract?"; a scenario that
drifts outside the contract catalog is a symptom of either an
undeclared contract or a superfluous test.

### 1.3 Profiles are fixtures with cost labels

Each workload profile (small/medium/large) carries an explicit cost
label: expected time budget, expected token budget, expected RAM
peak. Tests declare which profile they consume; the harness rejects
a test whose profile pick breaches the scope's duration budget.

### 1.4 No mocks at contract boundaries

A test that crosses an edge contract (E1-E14) may not mock the
transport layer. Contract tests use a real in-process transport (a
thread-safe channel for MCP, a temp SQLite file for the SQL mirror,
a real filesystem for jsonl). Mocking the transport invalidates the
contract test's purpose.

### 1.5 Scenario tests are generated, not handwritten

The scenario layer (§3.4) is generated from the adventure fixture +
a small set of scenario templates. Hand-written end-to-end tests
drift as the manifest schema evolves; generated ones update with the
schema. This is the autotest-orientation policy applied one level up.

## 2. Test Taxonomy

Five test kinds, each with an explicit purpose and non-goal. Every
test under `ark/tests/` is classified into exactly one of these.

### 2.1 Unit (per entity)

- **Purpose**: verify the behaviour of one entity's operations in
  isolation — schema parse, header set, event append, view render.
- **Scope**: one file under `ark/tests/test_<entity>_<kind>.py` or
  `ark/tests/<crate>/<module>.rs` per TC-019 §2.1 matrix.
- **Runtime budget**: < 100 ms per test; full unit suite < 30 s.
- **Non-goal**: cross-entity interactions, real I/O to external
  services.

### 2.2 Contract

- **Purpose**: verify that the producer and consumer of an edge
  contract (E1-E14) agree on schema, transport framing, and error
  envelope.
- **Scope**: one test per edge in `ark/tests/contracts/<edge>.py`,
  generated by Ark codegen from the Contract entity (T007 §10.4).
- **Runtime budget**: < 500 ms per contract; full contract suite
  < 30 s.
- **Non-goal**: multi-step scenarios, perf measurement.
- **Automation**: the codegen runs at the start of M8; a missing
  contract file fails CI.

### 2.3 Integration

- **Purpose**: verify that two or more entities interact correctly
  through their real ARL operations over a real substrate.
- **Scope**: one test per integration scenario under
  `ark/tests/integration/<scenario>.py`. Typical scenarios:
  - Writer/reader pair: `event_append + view_regen` round-trip.
  - Lock path: `task_start + task_complete` with a simulated second
    writer; asserts the lock is released on every path.
  - Stream replay: a pre-built 1 000-event stream is replayed; the
    reconstructed state equals a recorded snapshot.
  - Messenger approval: approval event round-trips from the agent,
    through the TM, to the lead's view.
- **Runtime budget**: < 10 s per test; full integration suite < 5 min.
- **Non-goal**: full adventure runs, real Anthropic calls.

### 2.4 E2E / Scenario

- **Purpose**: verify that a user-visible workflow completes across
  the full stack (TP + TM + Ark + optional BL mock) with real
  transports.
- **Scope**: one test per *scenario template* instantiated per
  profile. Scenarios:
  - `adventure.smoke` — a 3-task adventure from plan → implement →
    review → complete (T020 §M8).
  - `adventure.parallel` — two tasks in parallel, verifying no
    lock contention, no event interleaving.
  - `adventure.recovery` — inject a mid-task failure, assert the
    rollback path succeeds and the retry reaches completion.
  - `adventure.human_escalation` — force a lead escalation;
    verify messenger delivery and resume path.
- **Runtime budget**: < 2 min per scenario on small profile; < 6 min
  on medium; < 15 min on large.
- **Non-goal**: real Anthropic API (replaced with a cassette recorder);
  real github (replaced with a fake).

### 2.5 Chaos

- **Purpose**: exercise fault-injection paths — disk full, partial
  write, kill-9 mid-append, clock skew, duplicate event receipt.
- **Scope**: a dedicated harness under `ark/tests/chaos/` that wraps
  the E2E scenarios with a fault-injector.
- **Runtime budget**: chaos runs only nightly; full chaos suite < 30
  min on large profile.
- **Non-goal**: continuous PR gating — these are advisory, not
  blocking. A chaos-suite breach opens a GitHub issue but does not
  fail a PR unless explicitly tagged `chaos-gated`.

## 3. Scenario Catalog

The scenario layer (kinds 2.3, 2.4, 2.5) is where the bulk of the
post-reconstruction test value sits. This section enumerates the
scenarios by contract or entity pair.

### 3.1 Per-entity integration scenarios

Each entity (TC-019 §2.1) contributes at least one integration
scenario beyond its unit tests. The scenario demonstrates the
entity's ARL operations composed with at least one consumer.

| Entity | Scenario | Asserts |
|---|---|---|
| Task | `task.lifecycle_v2` | planning → implementing → completed transitions; stream events replay to final state |
| AdventureManifest | `manifest.targets_evaluations_sync` | append to targets/evaluations streams; view renders current state |
| AdventureLog | `log.role_attribution` | agent.emit populates attribution; no unattributed events |
| MetricsLedger | `metrics.sdk_source` | metrics pulled from SDK response, not proxy; aggregation across 100 events is monotonic |
| Lessons | `lessons.role_views` | role-scoped view fits ≤ 3 KB budget; newest-first ordering |
| Role | `role.spawn_identity` | spawned agent receives role prompt + auto-inject slice; no leaked context |
| Agent | `agent.permission_enforcement` | deny path returns structured error; allow path returns result |
| LeadState | `leadstate.race` | 50 concurrent emits cause zero lost events (T020 M3 gate) |
| Messenger | `messenger.approval_roundtrip` | approval event reaches lead view within budget; idempotent re-post |
| Permissions | `permissions.scope_load` | per-scope spec loads independently; no cross-scope pollution |
| Trigger | `trigger.hook_deterministic` | hook input → output is byte-equal over 100 runs (T020 M5 gate) |
| Config | `config.flag_flip` | feature flag flip reloads without restart |
| Run | `run.attribution` | every event carries run_id; rebuild from run_id reproduces state |
| Contract | `contract.version_mismatch` | TM refuses to start on incompatible consumer |
| Project | `project.boundary_resolution` | repo root detected; .agent path resolved; no walk-up past the project |

### 3.2 Per-edge contract scenarios (E1-E14)

Every edge from T006 §1.3 carries one contract test and at least one
integration scenario. The scenario list is keyed by edge:

| Edge | Integration scenario |
|---|---|
| E1 marketplace↔TM | `edge.e1_marketplace_discovery` — asserts auto-discovery lists exactly the active plugin versions |
| E2 TP↔.agent | `edge.e2_tp_agent_io` — asserts all writes go through ARL ops, no direct fs writes |
| E3 TM↔TP | `edge.e3_tm_tp_event_stream` — events flow TM→TP with bounded latency |
| E4 TM↔.agent | `edge.e4_atomic_writes` — asserts no partial file after kill-9 mid-write (H6) |
| E5 TP↔agents | `edge.e5_spawn_payload` — spawn payload size stays within token budget per tier |
| E6 messenger | `edge.e6_messenger_persist` — approval survives TM restart |
| E7 TM↔Ark | `edge.e7_tm_ark_codegen` — TM consumes generated registry |
| E8 BL↔TM | `edge.e8_bl_tm_ws` — WS frames preserve Union-typed payloads |
| E9 Ark↔Z3 | `edge.e9_ark_z3_proof` — verifier accepts canonical ARL, rejects off-canonical |
| E10 BL↔clients | `edge.e10_bl_client_session` — client reconnect replays from offset |
| E11 BL DSL | `edge.e11_bl_dsl_zod` — generated zod matches Ark schema byte-for-byte |
| E12 TP↔github | `edge.e12_github_workflow` — `run_workflow` invocation round-trips |
| E13 BL↔TP | `edge.e13_bl_tp_shared_lessons` — lessons written by BL visible in TP views |
| E14 Ark↔PDSL | `edge.e14_pdsl_dialect` — PDSL sample round-trips through Ark AST |

14 contract tests + 14 integration scenarios = 28 edge-level tests.

### 3.3 Full-stack E2E scenarios

| Scenario | Description | Profile | Runtime budget |
|---|---|---|---:|
| `adventure.smoke` | Single adventure, 3 tasks, green happy path | small | 2 min |
| `adventure.parallel` | 2 tasks in parallel, no contention | small | 3 min |
| `adventure.recovery` | Mid-task kill-9 then retry | medium | 5 min |
| `adventure.human_escalation` | Forced lead escalation | small | 3 min |
| `adventure.multi_role` | All 7 roles spawn within one adventure | medium | 6 min |
| `adventure.lesson_accrual` | 10 lessons accrue across 5 tasks; role views correct | medium | 4 min |
| `adventure.contract_drift` | Inject schema version mismatch; assert safe refusal | small | 2 min |
| `adventure.large_bench` | 20-task adventure; exercises benchmark fixtures | large | 15 min |

### 3.4 Scenario generator

Scenarios are emitted by an Ark-codegen template `scenario.ark` that
consumes:

1. A scenario skeleton (steps, expected transitions).
2. A profile fixture (workload size, seed, timeout).
3. A contract matrix (which edges the scenario exercises).

The generator emits one pytest file per (scenario × profile) pair.
Adding a new scenario is a spec edit + regen; no new Python is
handwritten. This is the autotest-oriented response to the "E2E tests
rot" complaint from phase3-1 role review.

### 3.5 Chaos scenarios

Chaos scenarios wrap E2E scenarios with fault injectors:

| Chaos wrapper | Applied to | Budget |
|---|---|---:|
| `chaos.fill_disk` | adventure.smoke | no data loss; graceful abort |
| `chaos.kill_mid_append` | adventure.parallel | recovered state ≡ pre-kill state + partial event discarded |
| `chaos.clock_skew` | adventure.recovery | event ordering preserved despite 30s skew |
| `chaos.duplicate_event` | adventure.smoke | idempotent handling; no double-apply |
| `chaos.slow_network` | adventure.multi_role | timeouts surface as typed errors, not crashes |
| `chaos.sdk_ratelimit` | adventure.large_bench | back-off respected; no budget overrun |

Six wrappers × three candidate scenarios each = ~18 chaos tests;
nightly only.

## 4. Workload Profiles

Three profiles: **small**, **medium**, **large**. Each profile is a
parameter vector consumed by fixture generators; the same seed yields
the same workload across runs.

### 4.1 Small profile

- 1 adventure, 3 tasks, 10 events per task.
- 2 roles active (planner + implementer).
- ~10 lessons accrued.
- No parallel spawns.
- Expected runtime: < 2 min E2E.
- Expected token cost: < 20 KB total.
- Expected RSS peak: < 200 MB.

**Use**: PR-gated scenarios, benchmark `op` scope, smoke.

### 4.2 Medium profile

- 1 adventure, 10 tasks, 50 events per task.
- 5 roles active.
- ~50 lessons accrued; role-view slicing non-trivial.
- Up to 2 parallel spawns at a time.
- Expected runtime: < 10 min E2E.
- Expected token cost: < 150 KB total.
- Expected RSS peak: < 400 MB.

**Use**: pre-merge scenarios, benchmark `stage` scope.

### 4.3 Large profile

- 1 adventure, 30 tasks, 200 events per task.
- All 7 roles active, overlapping.
- 200+ lessons; role-view slicing stress-tests ranking.
- Up to 8 parallel spawns.
- Simulated events span 90 days (for rollup views).
- Expected runtime: < 30 min E2E.
- Expected token cost: < 700 KB total.
- Expected RSS peak: < 800 MB.

**Use**: nightly scenarios, benchmark `adventure` scope, chaos.

### 4.4 Profile fixture shape

Every profile is a JSON file under `ark/tests/fixtures/profiles/`:

```
{
  "name": "medium",
  "seed": 42,
  "adventures": 1,
  "tasks_per_adventure": 10,
  "events_per_task": 50,
  "roles": ["lead", "planner", "researcher", "reviewer", "implementer"],
  "lesson_count": 50,
  "parallel_cap": 2,
  "event_time_span_days": 30
}
```

Profile files are versioned; a profile change requires a
`fixture_change` PR label (cf. benchmark §5.3).

## 6. Test Harness and Orchestration

### 6.1 Harness responsibilities

The `test` MCP tool (phase6 MCP-ops §2.4) is the single entrypoint
for running tests at any scope. Its responsibilities:

1. Select tests per `scope` and `filter`.
2. Provision the profile fixture (copy into temp dir, seed the RNG).
3. Spawn the subprocess (pytest or cargo test).
4. Capture stdout / stderr / timings into a structured envelope.
5. Emit `test.started` / `test.finished` events.
6. Return pass/fail with a structured error payload on failure.

The harness is language-agnostic: it wraps pytest and cargo test
behind a common result schema.

### 6.2 Scope taxonomy

Scopes extend the benchmark scope taxonomy:

| Scope | Contents | Duration budget |
|---|---|---:|
| `smoke` | `adventure.smoke` on small profile | 90 s |
| `unit` | all unit tests, no I/O | 30 s |
| `contract` | all 14 contract tests | 30 s |
| `integration` | all integration scenarios | 5 min |
| `e2e.small` | E2E scenarios on small profile | 5 min |
| `e2e.medium` | E2E scenarios on medium profile | 15 min |
| `e2e.large` | E2E scenarios on large profile | 30 min (nightly only) |
| `chaos` | all chaos wrappers | 45 min (nightly only) |
| `full` | unit + contract + integration + e2e.medium | 15 min |
| `bench` | benchmarks (see TC-024) | 15 min |

### 6.3 Parallel execution

The harness runs subprocess-level parallelism. Unit tests parallelise
by file; integration tests parallelise by scenario. E2E and chaos
tests run sequentially on large profile (to avoid OS-level contention
confounding results).

### 6.4 Failure triage

On any failure, the harness emits:

1. A structured `test.failed` event with the test id, subprocess exit
   code, and a pointer to the full stdout/stderr blob on disk.
2. A triage hint by matching the exit signature against a registry
   of known-failure signatures (e.g., "OOM", "port conflict", "SDK
   rate limit").
3. For integration/E2E failures, a replay bundle: the profile fixture
   + the event tail, packaged as a reproducer the implementer can
   run locally.

Failure triage cuts the reviewer's analysis time; the lead sees a
classified failure, not a wall of log output.

## 7. Integration with the Broader Pipeline

### 7.1 Spawned-agent impact

When a planner spawns, it receives the latest scope-summary view of
the test suite (green/red counts per subsystem, open autotest debt,
last bench-regression). This costs < 1 KB auto-inject per spawn and
gives the planner enough context to avoid scheduling a task whose
prerequisite suite is red.

### 7.2 Messenger approvals tied to test results

An implementer task that fails its scope's tests N times triggers a
messenger approval request to the lead before any further retry. The
test-harness failure count is the canonical input to this rule; the
messenger scope (per phase5) is the canonical delivery.

### 7.3 Knowledge events on test outcomes

Every red-to-green transition on a previously-failing test that has
been red ≥ 3 runs produces a `lesson.learned` event with
`kind: regression_fixed`, capturing the test id, the failing commits,
and the fixing commit. This feeds the knowledge base (M4) and
contributes to the regression-fixed view in role spawns.

### 7.4 CI coverage of this document's scenarios

Every scenario listed in §3 has a CI job enabled at the scope
specified in §6.2. The scope → scenario mapping is the canonical
index; reviewers consulting "does scenario X run in PR gating?"
check the map, not the CI YAML.

  scenario mapped in §3.1.
