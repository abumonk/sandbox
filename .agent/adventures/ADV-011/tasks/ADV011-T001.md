## Design

### Goal

Produce `.agent/adventures/ADV-011/research/concept-inventory.md` — a single flat markdown table that enumerates every distinct concept introduced by ADV-001..008 and ADV-010, tagged with the source adventure and the concrete artefact it came from. This file is the raw feed for ADV011-T003 (classification), ADV011-T004 (deduplication), and ADV011-T005 (pruning). It MUST NOT pre-classify, pre-merge, or pre-prune — it catalogs what exists.

### Table Schema (per `schemas/entities.md#ConceptInventoryRow`)

Exactly four columns, in this order. Header row must match TC-002 regex `^\| concept \| source_adventure \| source_artefact \| description \|` character-for-character:

| concept | source_adventure | source_artefact | description |
|---------|------------------|-----------------|-------------|

Column rules:
- `concept` — short noun phrase, kebab-or-Pascal form as written in the source (e.g., `Item::Expression`, `evolution_verify`, `Z3 ordinals for DAG acyclicity`, `telemetry metrics.md aggregator`). Non-empty. No commas inside pipes — escape with `\|` if needed.
- `source_adventure` — one value from `{ADV-001, ADV-002, ADV-003, ADV-004, ADV-005, ADV-006, ADV-007, ADV-008, ADV-010}`. Exactly one per row. If the same concept name appears in two adventures, emit two rows — dedup is T004's job, not T001's.
- `source_artefact` — concrete locator the reader can open. One of:
  - manifest section: `ADV-00N/manifest.md#Concept` or `#Target Conditions` or named section
  - stdlib file: `ark/dsl/stdlib/{name}.ark`
  - adventure report: `ADV-00N/reviews/adventure-report.md`
  - design doc: `ADV-00N/designs/{slug}.md`
  - research artefact: `ADV-00N/research/{slug}.md`
- `description` — ≤ 200 chars, single sentence, present tense. Quotes trimmed of the source's narrative voice.

### Traversal Rules (per adventure)

For each adventure directory, read in this order and extract concepts:

1. **`manifest.md#Concept`** — extract: every bolded term, every explicitly named DSL item kind (`Item::X`, `item X`), every named module (`X_verify`, `X_codegen`, `X_runner`), every named subsystem, every referenced Z3 pass or grammar extension.
2. **`manifest.md#Target Conditions`** (Description column only) — extract concepts introduced by TCs that were not already in `#Concept` (e.g., a TC referencing a specific struct name or pass id).
3. **stdlib `.ark` file(s)** listed per adventure above — extract each top-level declaration: `primitive X`, `struct X { ... }`, `enum X { ... }`, and any `item X`-style keyword. Only the declaration NAME and ITEM KIND go in the inventory; field lists do not. Comment-only concepts (e.g., "See tools/codegen/expression_primitives.py") are skipped — T001 records declared identifiers, not prose.
4. **`reviews/adventure-report.md`** (when present) — extract: any concept the report introduces that was not in the manifest (typically: integration-time discoveries, post-hoc module names, pattern names added to `.agent/knowledge/patterns.md`).
5. **`designs/*.md`** — read only when the manifest references them by slug and the concept is not derivable from manifest + stdlib (e.g., ADV-010's capture contract lives in a design, not the manifest). Do not enumerate every design doc exhaustively.

Files NOT to traverse in T001: test files, `plans/*.md`, per-task files, schema files, sibling-adventure artefacts. If the reader cannot find a listed expected concept in the sources above, omit the row and add a line to the task `## Log` stating what was not found and why — do not invent rows.

### Concept Granularity (what counts as one row)

To keep the inventory useful for downstream dedup, apply these granularity rules:

- **Item kind** (e.g., `Item::Expression`, `Item::StudioJob`, `Item::EvolutionOptimizer`) — one row per distinct kind.
- **Struct / enum / primitive declared in stdlib `.ark`** — one row per named declaration (`Vec3`, `Transform`, `GraphNode`, etc.). Math primitives in `types.ark` (Bool/Int/Float/...) are each one row; do not aggregate.
- **Runtime / tooling module** (e.g., `ark_verify.py`, `studio_codegen.py`, `AgentRunner`, `Watcher`) — one row per distinct module or orchestrator class named in the source.
- **Z3 pass / verifier rule** (e.g., `dag_acyclicity`, `opaque_primitive`, `numeric_interval`, `reference_exists`, `shape_termination`, `shape_determinism`) — one row per pass, identified by the pass-id token used in ADV-011 TC-011 where possible.
- **Research artefact** (ADV-007-heavy) — one row per named research deliverable slug (`synthesis-matrix.md`, `interaction-matrix.md`, `mv-sketches.md`, ...). The concept label is the artefact's subject, not its filename.
- **Grammar / AST extension** — one row per distinct grammar rule or AST node family introduced (e.g., `pipe_chain`, `predicate_combinator`, `studio_job_block`). Dual Lark+Pest counts as ONE concept (pattern "Dual-grammar parity"), not two.
- **Pattern imported from `.agent/knowledge/patterns.md`** is NOT a concept for T001 unless it is first introduced by one of these adventures. Patterns are catalogued by T003/T004.

### Duplicate Avoidance at Harvest Time

The manifest forbids duplication in the UNIFIED DESIGN (downstream), not in the inventory. T001 MUST preserve cross-adventure repetition because that signal is what T004 consumes to build the deduplication matrix. Concretely:

- If `Z3 ordinals` appears in ADV-003 and ADV-006, emit TWO rows (`source_adventure: ADV-003` and `source_adventure: ADV-006`) with identical or near-identical `concept` labels. Do not merge into one row with a list.
- If `telemetry` appears as a concept in ADV-001/002/005/006/010, emit FIVE rows.
- Within a single adventure block, however, do NOT emit the same `(concept, source_artefact)` pair twice. If a concept is mentioned in both `#Concept` and `#Target Conditions` of the same manifest, prefer `#Concept` as the `source_artefact` (more stable locator) and skip the second mention.
- Exact-duplicate detection rule: two rows are considered a within-block duplicate iff `(source_adventure, concept.lower(), source_artefact)` match. Pre-submission sanity check: sort rows by the triple and assert no equal adjacent rows.

### Risks

- **R1 — ADV-007 bloat**: ADV-007 has ~20 research artefacts; naively enumerating each may push the inventory past 150 rows and make T003 mapping tedious. Mitigation: one row per *named deliverable slug*, not per section within a deliverable. T003 will bulk-classify ADV-007's block as `out-of-scope: ecosystem` — inventory granularity there can stay coarse.
- **R2 — ADV-010 has no adventure-report**: state is `review`. Mitigation: traversal rules above already skip that artefact for ADV-010 and cite manifest + designs instead. Reviewer should not flag the absence.
- **R3 — Overlap between stdlib items and item kinds**: e.g., `agent.ark` may declare both `struct AgentRunner { ... }` (struct) and imply item kind `Item::Agent`. Mitigation: emit BOTH rows (one as struct, one as item-kind), each with its own `source_artefact`. Granularity rule makes this unambiguous.
- **R4 — Pattern vs concept confusion**: `Dual-grammar parity` is a pattern; `ark.pest` + `ark_grammar.lark` together are ONE concept. Mitigation: granularity rule 6 above — one row per grammar extension, not per grammar file. Patterns are excluded entirely from T001.
- **R5 — TC-002 regex is character-exact**: a missing space or extra pipe will fail the autotest. Mitigation: copy the header line verbatim from the manifest's TC-002 command string into the file.
