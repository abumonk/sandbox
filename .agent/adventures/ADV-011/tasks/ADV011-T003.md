## Description
Consume `research/concept-inventory.md`. For each concept, apply the classification rules from `design-concept-mapping.md` and record a row in `research/concept-mapping.md`. Every row carries exactly one bucket tag. Any multi-source concept gets a single row listing all sources in `source_adventure`. Include a `## Per-Bucket Rationale` trailing section (4 paragraphs).

## Acceptance Criteria
- File exists with master table + per-bucket rationale section.
- Every inventory row is represented in the mapping (enforced by T011 test_mapping_completeness).
- Every bucket tag comes from the fixed set of four.

## Design

### Goal

Produce `.agent/adventures/ADV-011/research/concept-mapping.md` — a single markdown file containing (a) a master classification table with one row per concept drawn from `research/concept-inventory.md` (ADV011-T001 output) and (b) a trailing `## Per-Bucket Rationale` section with exactly four paragraphs, one per bucket. Every row's `bucket` cell must hold exactly one of `{descriptor, builder, controller, out-of-scope}`. This file is the structural precursor for dedup (T004), pruning (T005), and the three unified-design deltas (T006/T007/T008).

### Input Contract

Read `.agent/adventures/ADV-011/research/concept-inventory.md` (produced by T001). That file's rows are the *only* source of concepts to classify — do NOT re-harvest from manifests or stdlib. Rows are keyed on the 4-column schema from `schemas/entities.md#ConceptInventoryRow`:

```
| concept | source_adventure | source_artefact | description |
```

T001's duplicate-preservation rule means the same concept label may appear in multiple inventory rows under different `source_adventure` values (e.g., "Z3 ordinals" appears under ADV-003 and ADV-006; "telemetry" under ADV-001/002/005/006/010). T003 MUST collapse those into a single mapping row whose `source_adventure` cell lists all source adventures comma-separated (e.g., `ADV-003, ADV-006`). This is the first deduplication pass in the pipeline; T004 will then attach canonical_form and unification_action on top of this collapsed view.

### Output Table Schema (per `schemas/entities.md#ConceptMappingRow`)

Exactly five columns, in this order:

| concept | source_adventure | bucket | canonical_name | notes |
|---------|------------------|--------|----------------|-------|

Column rules:
- `concept` — copied verbatim from the inventory (for multi-source concepts, pick the label used by the earliest-numbered source adventure; record any variant spellings in `notes`).
- `source_adventure` — one adventure id OR a comma-separated list of ids (e.g., `ADV-003, ADV-006`). The list must be sorted ascending by adventure number. Every id in the list must correspond to a row in `concept-inventory.md` carrying the same concept.
- `bucket` — exactly one of `descriptor | builder | controller | out-of-scope`. No other value. No multi-bucket rows. This is the column enforced by TC-004 and the `test_buckets_allowlist` unittest (T011).
- `canonical_name` — short post-dedup name in the form the unified design will reference. For now, a best-effort normalization (e.g., `Item::Expression` → `Item::Expression`; `Z3 ordinals` + `Z3 ordinals for DAG acyclicity` → `z3_ordinal_pass`). T004 may override this with a stronger canonical_form; that is acceptable and does not invalidate T003.
- `notes` — optional, ≤ 200 chars. Use for: variant spellings across sources, classification hint for T004/T005, or pointer to the future adventure slug if `bucket == out-of-scope`.

The master table MUST be preceded by a 3–5 line preamble stating: purpose, schema reference (`schemas/entities.md#ConceptMappingRow`), bucket allowlist (exact set of four strings), and a pointer to the rationale section below.

### Classification Decision Rules Per Bucket

Apply these rules in order; the first rule that matches wins. A concept's bucket is determined by the thing the concept names, not by which adventure introduced it.

**1. Descriptor** — assign if the concept is any of:
   - A DSL item kind (`Item::Expression`, `Item::Predicate`, `Item::StudioJob`, `Item::EvolutionOptimizer`, visual/agent/evolution item kinds, etc.).
   - A struct / enum / primitive declared in `ark/dsl/stdlib/*.ark` (e.g., `Vec3`, `Transform`, `GraphNode`, math primitives from `types.ark`).
   - A grammar rule or AST node family (`pipe_chain`, `predicate_combinator`, `studio_job_block`, `Dual-grammar parity` as a pattern anchor).
   - A parser artefact (`ark.pest`, `ark_grammar.lark`, schema structs derived from `.ark`).
   - A parameter/reference form (`@var, [prop], #idx, {nested}`), predicate combinator (`|AND|OR|XOR|!|{}`), or any concept whose role is "how the user *writes* a spec".
   The descriptor bucket answers: *"is this part of what a user types or the schema the parser produces?"*

**2. Builder** — assign if (and descriptor did not match) the concept is any of:
   - A verifier / Z3 pass (`dag_acyclicity`, `opaque_primitive`, `numeric_interval`, `reference_exists`, `shape_termination`, `shape_determinism`, `expression_smt`, `visual_ordinal`).
   - A codegen module (`ark_codegen`, `studio_codegen`, `evolution_codegen`, `agent_codegen`, `visual_codegen`, `expression_primitives`).
   - A verification infrastructure module (`ark_verify`, `studio_verify`, `evolution_verify`, `agent_verify`, `visual_verify`, `graph_verify`).
   - A static-analysis / query / indexing tool operating on descriptors (`GraphStore`, `CodeIndexer`, `QueryEngine`, `Watcher`, impact/diff tools).
   - A visualiser that renders a *built* artefact (mermaid/Pillow renderers, diagram codegen) — but NOT a live runtime UI surface (those are out-of-scope).
   - A `PASS_OPAQUE` usage or opaque-primitive-tracking facility.
   The builder bucket answers: *"does this transform, verify, or analyse a descriptor to produce a checked/executable artefact?"*

**3. Controller** — assign if (and neither above matched) the concept is any of:
   - A runtime orchestrator (`AgentRunner`, gateway, scheduler, evaluator, skill-manager).
   - A runtime backend / executor (agent backends, learning loop, self-evolution loop).
   - A telemetry / observability component (ADV-010: metrics.md row schema, SubagentStop hook, SDK usage fields, cost model, idempotent append, aggregator, task-actuals pipeline, metrics-capture-error entry, backfill mechanism).
   - An event contract or runtime state model element.
   - A reflexive/dogfooding runtime surface (`evolution_skills.ark` consumed at runtime, reflexive self-indexing of the repo).
   - A review / human-in-the-loop runtime component.
   The controller bucket answers: *"does this run, observe, or adapt a built system at execution time?"*

**4. Out-of-scope** — assign if none of the above matched, OR if the concept is explicitly one of:
   - A UI / Electron / screenshot / visual-runtime surface that is not a codegen output (e.g., ADV-006 visual screenshot surfaces as interactive UI).
   - An ADV-007 ecosystem roadmap artefact (MCP tier catalogue, synthesis matrix, interaction matrix, MV sketches, growth-path docs, forward-only trapdoors, append-only jsonl substrate, abstract storage algebra, lightweight_index scalar, file→directory promotion). Per the manifest, ADV-007's research block collapses to out-of-scope with an `ecosystem` tag in `notes`, but each inventory row must still emit its own mapping row (collapsed rationale lives in the per-bucket paragraph, not via row-merging).
   - A sibling-package semantics note from ADV-008 that is about the *host-language contract* rather than ark-core DSL (e.g., `lazy-import circular-dep break`).
   - A speculative / research-only idea not tied to any of the three core roles.
   - A doc generator, ecosystem tooling, or non-core CLI.
   Every out-of-scope row's `notes` cell MUST carry either `OUT-OF-SCOPE -> ADV-<NN>` (pointer to a plausible future adventure, same vocabulary as `PruningCatalogRow.disposition`) OR `DROP` (concept retired with no successor). T005 will later consume this field to build `pruning-catalog.md`.

**Tiebreak rule**: if a concept could plausibly sit in two buckets (e.g., "predicate combinator" might look like both descriptor *and* builder because the combinator is implemented in codegen), pick the bucket whose question the concept primarily answers from the user's perspective. Record the alternative bucket in `notes` prefixed `alt:` so T004 sees it.

### Handling Multi-Source Concepts

Inventory rows with the same `concept` label under different `source_adventure` values (T001 preserves these deliberately) collapse to a single mapping row per the following procedure:

1. Group inventory rows by `concept.lower()` (case-insensitive) AND semantic equivalence (e.g., `Z3 ordinals` and `Z3 ordinals for DAG acyclicity` are semantically one concept — treat as same group). Record the variant spellings in the row's `notes` cell.
2. For the collapsed row, `source_adventure` becomes the comma-separated sorted list of adventure ids from the group.
3. `canonical_name` is the cleaner / more generic of the variant labels (typically the one without adventure-specific suffix).
4. All rows in the group must receive the **same bucket** — if the classification rules yield different buckets across group members, that is a signal the group was wrongly merged; split and re-classify.
5. Expected known collapsed groups (from the manifest's dedup seed list — T004 will validate):
   - `Z3 ordinals` — sources ADV-003, ADV-006 → bucket: builder.
   - `Dual-grammar parity (Lark + Pest)` — sources ADV-001 plus any adventure that extends the grammar → bucket: descriptor.
   - `telemetry` — sources ADV-001, ADV-002, ADV-005, ADV-006, ADV-010 → bucket: controller.
   - `PASS_OPAQUE usage` — sources ADV-001 plus any adventure re-using it → bucket: builder.
   - `reflexive dogfooding` — sources ADV-002 (self-indexing), ADV-004 (`evolution_skills.ark`) → bucket: controller (with `alt: descriptor` note where applicable).
   - `Skill definition` — sources wherever Skill structs are declared → bucket: descriptor.

### Per-Bucket Rationale Section Structure

The file ends with a section headed exactly `## Per-Bucket Rationale` (this exact string is what TC-003's grep tests). Under that heading, four paragraphs follow in fixed order, each introduced by a bolded bucket name on its own line so they are visually separable:

```
## Per-Bucket Rationale

**Descriptor.**
<one paragraph, 4-8 sentences>

**Builder.**
<one paragraph, 4-8 sentences>

**Controller.**
<one paragraph, 4-8 sentences>

**Out-of-scope.**
<one paragraph, 4-8 sentences>
```

Each paragraph must:
- State in plain prose what the bucket *is* (the question it answers).
- Cite 2–4 representative concepts drawn from the master table above (by name, matching the `concept` column exactly) to ground the abstraction.
- State the classification boundary that distinguishes this bucket from its nearest neighbours (descriptor↔builder boundary, builder↔controller boundary, in-core↔out-of-scope boundary).
- For the out-of-scope paragraph: explicitly name the ADV-007 ecosystem block as the dominant cause of volume and state that those rows collapse into the pruning catalog (T005).
- Be 4–8 sentences. A paragraph of fewer than 4 sentences will be flagged by reviewer; more than 8 suggests it belongs in a design doc, not a rationale.

### Invariants the T011 Completeness Test Will Check

- **I1 — File exists**: `test -f .agent/adventures/ADV-011/research/concept-mapping.md` (covered by TC-003).
- **I2 — Section exists**: the literal string `## Per-Bucket Rationale` appears exactly once in the file (covered by TC-003).
- **I3 — Bucket allowlist (TC-004, `test_buckets_allowlist`)**: every non-header, non-separator row in the master table has its `bucket` cell match the regex `^(descriptor|builder|controller|out-of-scope)$` — case-sensitive, no surrounding whitespace beyond the cell padding. Parser: split each non-header row on `|`, strip whitespace, index the bucket column (index 3 counting from column 1 after the leading pipe), assert membership in the 4-element allowlist.
- **I4 — Coverage (`test_mapping_completeness`)**: for every inventory row in `concept-inventory.md`, at least one mapping row exists whose `concept` matches AND whose `source_adventure` cell contains that inventory row's `source_adventure`. Equivalent formulation: for each `(concept, source_adventure)` pair in the inventory, there exists some mapping row with that concept whose `source_adventure` list (split on comma, whitespace-trimmed) contains that adventure id. No inventory row may be silently dropped.
- **I5 — No duplicate mapping rows**: no two mapping rows share the same `canonical_name`. If two rows share a canonical_name they should have been collapsed.
- **I6 — Out-of-scope disposition populated**: every row with `bucket == out-of-scope` has a non-empty `notes` cell matching `^(OUT-OF-SCOPE -> ADV-[0-9]{3}|DROP)( .*)?$`. (This invariant is advisory at T003 — T005 will enforce strictly via TC-008 against the pruning catalog; T003 only needs to not block that downstream check.)
- **I7 — Four rationale paragraphs**: under `## Per-Bucket Rationale`, exactly four paragraph blocks appear, each preceded by one of `**Descriptor.**`, `**Builder.**`, `**Controller.**`, `**Out-of-scope.**` in that order. A cheap check is: `grep -E '^\*\*(Descriptor|Builder|Controller|Out-of-scope)\.\*\*$' | wc -l == 4`.
- **I8 — Header regex**: the master table's header row matches `^\| concept \| source_adventure \| bucket \| canonical_name \| notes \|` exactly. Column count = 5.

T011 will implement at minimum `test_buckets_allowlist` (I3) and `test_mapping_completeness` (I4) as separate `unittest.TestCase` methods; other invariants may be folded into those methods or left as grep-level checks in `run-all.sh`.

### Implementation Steps

1. Open `.agent/adventures/ADV-011/research/concept-inventory.md` and parse its rows into memory (conceptually — this is a manual authoring task, not a script).
2. Group inventory rows into candidate mapping rows per the multi-source collapse rules above.
3. For each candidate group, apply the 4-rule classification decision ladder; record the bucket and a canonical_name.
4. Write the preamble (3–5 lines) to `research/concept-mapping.md`.
5. Write the 5-column header row matching I8 verbatim + the separator row.
6. Emit mapping rows grouped by bucket in order `descriptor` → `builder` → `controller` → `out-of-scope` (this ordering is not enforced by any TC but makes review easier and matches the rationale section order).
7. Append the `## Per-Bucket Rationale` section with four structured paragraphs per the template above.
8. Self-check invariants I1–I8 manually before finalizing.
9. Append to task `## Log` a summary: row counts per bucket, count of collapsed multi-source rows, any classification judgement calls worth flagging.

### Risks

- **R1 — Descriptor↔builder edge cases**: concepts like "predicate combinator" or "parameter ref syntax" can look like both schema and codegen. Mitigation: apply the tiebreak rule (user's perspective wins) and record the alternative in `notes` prefixed `alt:`.
- **R2 — ADV-007 ecosystem row explosion**: likely 15–25 rows all landing in out-of-scope. Mitigation: emit one mapping row per inventory row (do NOT pre-collapse ADV-007 rows into a single synthetic row — the granularity must survive to T005's pruning catalog). The per-bucket rationale paragraph handles the narrative collapse.
- **R3 — Multi-source label mismatch**: ADV-003 may spell it `Z3 ordinals` while ADV-006 spells it `Z3 ordinal pass`. Mitigation: semantic-equivalence grouping (step 1 of multi-source handling) plus variant-spelling record in `notes`.
- **R4 — Bucket allowlist string exactness**: `out-of-scope` must have lowercase `o`, hyphens not underscores, no trailing whitespace. The unittest matches exact bytes. Mitigation: type each of the 4 strings once into a preamble constant list and copy into rows.
- **R5 — Rationale paragraph word-count drift**: over-short (<4 sentences) or over-long (>8) will draw review friction. Mitigation: draft paragraphs against a sentence counter; use the 4-sentence minimum as a hard floor.
- **R6 — Coverage loss from aggressive merging**: collapsing two inventory rows into one mapping row is safe only if their adventure ids both end up in `source_adventure`. Mitigation: after collapse, enumerate every `(concept, adventure)` pair in the inventory and grep the mapping file to confirm each is discoverable. This is exactly what I4 / `test_mapping_completeness` will automate.
