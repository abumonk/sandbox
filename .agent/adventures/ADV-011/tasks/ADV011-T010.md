## Description
Produce `research/downstream-adventure-plan.md` with 3–6 downstream adventures per `design-downstream-adventure-plan.md`. Seed list (ADV-DU, ADV-BC, ADV-CC, ADV-OP, optional ADV-CE and ADV-UI). Each adventure sketch has Concept / Scope / Depends on / Est. task count. Document the serial-ordering constraint `ADV-DU → ADV-BC → ADV-CC → ADV-OP`.

- File `.agent/adventures/ADV-011/research/downstream-adventure-plan.md` exists.
- Between 3 and 6 H2 sections whose headings match `^## ADV-[A-Z]{2}\b` (TC-017: `grep -cE "^## ADV-" … ` yields 3..6 inclusive).
- Every H2 adventure section contains all four required fields as H3 (or labelled bold lines): `Concept`, `Scope`, `Depends on`, `Est. task count`.
- Serial ordering constraint stated explicitly in a line matching the TC-018 regex `ADV-DU.*ADV-BC.*ADV-CC.*ADV-OP` (single-line; arrows must appear in that left-to-right order on one line).
- Every `DEFERRED-TO: ADV-XX` id that appears in `research/validation-coverage.md` (produced by T009) is present as an H2 adventure heading in this plan — no dangling deferrals.
- ADV-DU, ADV-BC, ADV-CC, ADV-OP are all present (mandatory core four). ADV-CE and ADV-UI appear iff admission rules in the design are met; otherwise they are explicitly listed in an "Excluded (not scheduled)" subsection with rationale.

## Design

Summary: the research note produced by this task is a structured roadmap of
3–6 downstream implementation adventures. The four mandatory adventures
(ADV-DU, ADV-BC, ADV-CC, ADV-OP) form a strictly serial chain; optional
adventures ADV-CE and ADV-UI are admitted only when the T009 validation
matrix or T005 pruning catalog creates demand for them. The serial ordering
is stated in a single canonical sentence whose regex is codified by TC-018.

See: `.agent/designs/ADV011-T010-design.md` for the full spec (H2 schema,
required fields, admission rules, ordering sentence, TC mapping).

### H2 section schema (one per adventure)

Each adventure sketch MUST be introduced with an H2 of the exact form:

```
## ADV-<CODE> <Human Title>
```

where `<CODE>` is one of `DU`, `BC`, `CC`, `OP`, `CE`, `UI` (two uppercase
letters; matches `TC-017` counter `^## ADV-`). Example:

```
## ADV-DU Descriptor Unification
```

Within each H2 section, the four required fields appear in this fixed order
using H3 headings (labelled-bold lines are also permitted but H3 is
preferred for stable anchoring):

```
### Concept
<one paragraph: what the adventure accomplishes and what artefacts it
touches, in core-role language — descriptor / builder / controller.>

### Scope
<bullet list (3–8 bullets) naming the directories, modules, stdlib files,
or artefact classes the adventure modifies. Explicitly call out
"No new capabilities." or list allowed new capabilities if any.>

### Depends on
<list of upstream ADV-XX ids from this plan, or `none`. Must be consistent
with the serial-ordering constraint below.>

### Est. task count
<integer or small range like "8–12". Must respect ≤20 tasks/adventure
envelope from the overarching design. Include one-line rationale.>
```

No other H3 subsections are required. Optional `### Notes` / `### TCs implied`
may follow but are not scored.

### Serial-ordering constraint — canonical sentence

A single H2 section `## Ordering Constraint` MUST contain, on one physical
line, the exact sentence:

```
The four mandatory adventures are strictly serial: ADV-DU → ADV-BC → ADV-CC → ADV-OP.
```

This satisfies TC-018 (`grep -qE "ADV-DU.*ADV-BC.*ADV-CC.*ADV-OP"`). The
writer MUST use the en-dash-free ASCII `->` only if unicode `→` is not
available; the regex tolerates either because it matches only the ids and
intervening characters. The recommended form uses `→` to match the
manifest and the wave-E plan wording.

A second line in the same section states the rules for optional adventures:

```
ADV-CE and ADV-UI, if admitted, are scheduled after ADV-OP in any order.
```

### Rules for deciding 3-vs-6 count (admission matrix)

The final count is driven by the T009 validation matrix and the T005
pruning catalog, in this priority order:

1. **Floor = 4** — ADV-DU, ADV-BC, ADV-CC, ADV-OP are ALWAYS present.
   They are the realisation adventures of the three unified designs
   plus the out-of-scope parking step required by the pruning catalog.
   Count cannot go below 4.

2. **ADV-CE admitted iff**:
   - `research/validation-coverage.md` contains at least one row with
     `DEFERRED-TO: ADV-CE`, OR
   - `research/pruning-catalog.md` contains an `OUT-OF-SCOPE → ADV-CE`
     disposition row for a Hermes / code-evolution artefact.
   Otherwise ADV-CE is excluded and listed in the "Excluded (not
   scheduled)" subsection with a one-line rationale referencing the
   pruning catalog row that dropped evolution ("DROP" disposition).

3. **ADV-UI admitted iff**:
   - `research/validation-coverage.md` contains at least one row with
     `DEFERRED-TO: ADV-UI`, OR
   - `research/pruning-catalog.md` contains at least one
     `OUT-OF-SCOPE → ADV-UI` disposition row for a UI-adjacent artefact
     (screenshot_manager, visual_search advanced modes, html_previewer
     advanced modes, Electron surfaces).
   Otherwise excluded with rationale.

4. **Ceiling = 6** — the plan MUST NOT introduce adventures beyond
   {DU, BC, CC, OP, CE, UI}. If T009 produces a `DEFERRED-TO: ADV-XX`
   for some unrelated id, the writer escalates (does not silently
   invent a seventh adventure) and logs a planning gap for the lead.

5. **Counting rule for TC-017**: only H2 lines matching `^## ADV-` are
   counted. The `## Ordering Constraint` H2 and any other narrative H2
   headings do NOT start with `ADV-` and therefore do NOT inflate the
   count.

### Excluded adventures subsection

If either optional adventure is excluded, the plan ends with:

```
## Excluded (not scheduled)

- **ADV-CE Code Evolution** — <one-line rationale citing the T009/T005
  row that justifies exclusion; e.g., "all Hermes evolution artefacts
  carry DROP disposition in pruning-catalog rows 4–6; no DEFERRED-TO:
  ADV-CE in validation-coverage.md">
- **ADV-UI UI Sibling** — <same pattern>
```

This subsection is narrative; it does NOT add H2 `## ADV-` lines and
therefore does not affect TC-017 counting.

### Dependency consistency rules (internal)

Within each admitted adventure's `### Depends on` section, the writer
checks:

- ADV-DU: `none`.
- ADV-BC: `ADV-DU`.
- ADV-CC: `ADV-DU, ADV-BC`.
- ADV-OP: `ADV-DU, ADV-BC, ADV-CC`.
- ADV-CE (if admitted): `ADV-CC` (requires controller runtime).
- ADV-UI (if admitted): `none` (sibling package; does not depend on
  the core chain, but is scheduled after ADV-OP per the ordering
  constraint to avoid splitting reviewer attention).

Any deviation must be justified in a `### Notes` subsection.

### Out of scope for this task

  controller designs — those exist under `designs/` and are referenced,
  not duplicated.
- Task decomposition of each downstream adventure. Only the `Est. task
  count` integer/range is required.
- Budget calculations — the overarching design already states the
  "≤20 tasks at sonnet rates, $4–8 per adventure" envelope; this plan
  may cite it but need not re-derive it.
