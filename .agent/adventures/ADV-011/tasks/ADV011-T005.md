## Description
Produce `research/pruning-catalog.md` per `design-pruning-catalog.md`. Drive from the `bucket = out-of-scope` rows of `concept-mapping.md`. Seed the catalog with the 7 candidates from the adventure brief (Darwinian stub, screenshot/visual_search/html_previewer, ecosystem artefacts, master roadmap, shape_grammar content, Darwinian git-organism, MCP server catalog). Every row carries `justification (≥ 40 chars)`, `disposition`, `owner_risk`. Disposition is one of `OUT-OF-SCOPE → ADV-NN` or `DROP` — no other values permitted.

## Acceptance Criteria
- File exists.
- Every out-of-scope row from the mapping appears in the catalog.
- Every `OUT-OF-SCOPE → ADV-NN` disposition names an adventure that will be listed in T010's downstream plan.
- Every `DROP` disposition carries a justification ≥ 40 chars.

## Design

See `.agent/adventures/ADV-011/designs/design-pruning-catalog.md` for the
over-arching design. This section nails down the *row-level* schema, the
disposition allowlist enforcement, the forward-reference protocol that binds
each `OUT-OF-SCOPE → ADV-NN` row to T010's downstream plan, and a previewed
row for each of the 7 seed candidates so the implementer can copy-paste into
`research/pruning-catalog.md`.

### Approach

1. Open `research/concept-mapping.md` produced by T003 and extract every row
   with `bucket = out-of-scope`.
2. For each extracted row, author a `PruningCatalogRow` per the schema below.
3. Ensure at least the 7 seed candidates below are present (the brief
   mandates 7; TC-007 asserts ≥ 9 table rows counting the header and separator
   rows, so the implementer must emit ≥ 7 data rows in addition to the two
   header lines — see "TC-007 row-count accounting" below).
4. Run the disposition allowlist check mentally before save: every value in
   the `disposition` column must match the regex
   `^(OUT-OF-SCOPE → ADV-[A-Z0-9-]+|DROP)$`.
5. Record per-row `owner_risk` so the unified-design readers understand what
   capability is forfeited by pruning.

### Catalog file skeleton

The file `research/pruning-catalog.md` must open with:

```
# Pruning Catalog — ADV-011

Generated from `research/concept-mapping.md` rows where `bucket = out-of-scope`.
Seed rows from the adventure brief (ADV-011 manifest §What this adventure delivers).

## Table

| concept | source_adventure | justification | disposition | owner_risk |
|---------|------------------|---------------|-------------|------------|
| ... one row per pruned concept ... |
```

…followed by a `## Per-Row Notes` section keyed by concept for anything that
does not fit in the cell (justification overflow, links to future adventure
titles, cross-references to dedup-matrix rows that subsume this concept).

### Column schema (PruningCatalogRow)

Matches `schemas/entities.md §PruningCatalogRow`. All fields are MANDATORY.

| column | type | constraint |
|--------|------|-----------|
| `concept` | string | non-empty; matches the concept name used in `concept-mapping.md` exactly so the cross-check grep in T011 passes |
| `source_adventure` | enum | one of `ADV-001..008` or `ADV-010` |
| `justification` | string | **≥ 40 characters** (excluding pipes and surrounding whitespace). Must state *why* the concept is not descriptor/builder/controller, not merely restate what it is. |
| `disposition` | enum | exactly `OUT-OF-SCOPE → ADV-NN` (where `ADV-NN` is a placeholder adv-id from T010) OR `DROP` |
| `owner_risk` | string | names the consumer that breaks if pruned without replacement; `"none"` is an allowed value when no core caller depends on the concept |

Cell escaping: if any cell contains a literal `|`, escape as `\|`. Table rows
must not wrap; long justifications should be truncated in the cell with a
trailing `…` and the full text placed in `## Per-Row Notes`.

### Disposition allowlist enforcement

TC-008 encodes the allowlist as:

```
! grep -E "^\|.*\|.*\|.*\|.*\|.*\|" pruning-catalog.md \
  | grep -vE "(OUT-OF-SCOPE|DROP|disposition|---)"
```

Implementer-facing rules to guarantee this passes:

1. Only two literal disposition shapes are permitted in the `disposition`
   column: `OUT-OF-SCOPE → ADV-NN` or `DROP`. **No** synonyms (`out of scope`,
   `drop it`, `TBD`, `REVIEW`, `defer`, `OUT-OF-SCOPE` without the arrow and
   target, etc.) are allowed — they will fail TC-008.
2. The literal `→` is a U+2192 RIGHTWARDS ARROW, not `->`. File must be UTF-8
   without BOM.
3. Every `ADV-NN` placeholder must be a concrete identifier (the four
   sketches in T010 use `ADV-DU`, `ADV-BC`, `ADV-CC`, `ADV-OP` — see forward
   reference protocol below). If the correct target is not yet known at
   T005 write-time, use one of the four canonical downstream ids in the
   disposition and record the forward reference in `## Per-Row Notes`.
4. When a `DROP` row is emitted, the justification column MUST contain ≥ 40
   chars and state why nothing downstream needs the concept (acceptance
   criterion 4 of this task).

### Forward-reference protocol (binding to T010)

Every `OUT-OF-SCOPE → ADV-NN` row asserts a forward reference that T010 must
honour. Protocol:

1. T005 chooses `ADV-NN` from the **canonical downstream id set** published
   in `designs/design-downstream-adventure-plan.md` §"Canonical downstream
   ids": `{ADV-DU, ADV-BC, ADV-CC, ADV-OP, ADV-UI, ADV-CE}` where
   - `ADV-DU` = Descriptor Unification
   - `ADV-BC` = Builder Consolidation
   - `ADV-CC` = Controller Consolidation
   - `ADV-OP` = Operational Polish
   - `ADV-UI` = (provisional) Ark UI surface — hosts pruned Electron/Pillow/Ollama work
   - `ADV-CE` = (provisional) Ark Code Evolution — hosts Darwinian git-organism
2. T005 may reference the provisional ids `ADV-UI` and `ADV-CE`. T010 is
   expected to emit them as sketches (or promote them to numbered downstream
   adventures) so the TC-007 cross-check holds. If T010 decides not to
   materialise `ADV-UI` or `ADV-CE`, it MUST rewrite the affected rows in
   `pruning-catalog.md` to `DROP` during that task, not leave dangling
   references.
3. The bookkeeping row in `## Per-Row Notes` for each forward reference must
   cite the downstream adventure title (human-readable) so a reviewer of the
   pruning catalog can follow the trail without opening `downstream-adventure-plan.md`.
4. T009 (`research/validation-coverage.md`) will reverse-check this protocol:
   every `OUT-OF-SCOPE → ADV-NN` disposition corresponds to a
   `DEFERRED-TO ADV-NN` verdict.

### TC-007 row-count accounting

TC-007 proof command:

```
[ $(grep -cE "^\|.*\|" pruning-catalog.md) -ge 9 ]
```

The regex `^\|.*\|` matches any Markdown table line — header, separator, and
data rows all count. With ≥ 7 data rows + 1 header row + 1 separator row the
count is ≥ 9. The implementer MUST NOT add extra `|`-prefixed lines outside
the single catalog table (e.g. no sub-tables under `## Per-Row Notes` that
use pipes), otherwise the count inflates and hides missing data rows.

### Implementation Steps

1. Read `research/concept-mapping.md` and list every `bucket = out-of-scope`
   row; sanity-check the 7 seed concepts are all present under that bucket.
   If one is missing, the mapping (T003 output) is wrong — raise as a
   blocker rather than synthesise a row unilaterally.
2. Create `research/pruning-catalog.md` with the skeleton shown above.
3. Transcribe the 7 seed rows from the previews above, adjusting
   `ADV-NN` placeholders only if T010 has already renamed the canonical
   downstream ids (unlikely at T005 write-time).
4. For each additional `out-of-scope` row from the mapping that is *not* one
   of the 7 seeds, author a new row following the same schema. If a mapping
   row's concept is covered by the deduplication matrix with an assigned
   bucket of `descriptor|builder|controller`, it MUST NOT appear here — it
   is unified, not pruned.
5. Populate `## Per-Row Notes` for every `OUT-OF-SCOPE → ADV-NN` row with a
   one-line pointer to the future adventure title.
6. Self-check against TC-007 (`grep -cE "^\|.*\|"` ≥ 9) and TC-008
   (disposition allowlist) before committing.

### Testing Strategy

- TC-007: counted via `grep -cE "^\|.*\|" research/pruning-catalog.md` ≥ 9.
- TC-008: no disposition-column cell falls outside the allowlist — enforced
  by the negated grep shown above.
- Task-local acceptance: manually audit that every justification string is
  ≥ 40 chars (Python one-liner acceptable:
  `python -c "import re,sys; [print(len(c.strip())) for c in re.findall(r'\|([^|]+)\|', open('research/pruning-catalog.md',encoding='utf-8').read())]"`).
- Forward-reference consistency: T009 will cross-check; any
  `OUT-OF-SCOPE → ADV-NN` without a corresponding `DEFERRED-TO ADV-NN`
  verdict in `validation-coverage.md` is a T009 defect, not a T005 defect.

### Risks

- **T003 mapping is incomplete**: if one of the 7 seed concepts is not in
  `concept-mapping.md` with `bucket = out-of-scope`, T005 cannot pass
  acceptance criterion 2 ("every out-of-scope row from the mapping appears
  in the catalog") without synthesising. Mitigation: block on T003 revision
  rather than paper over.
- **Downstream id drift**: T010 might rename `ADV-UI` or `ADV-CE`. Mitigation:
  T010's planner owns rewriting any affected `OUT-OF-SCOPE → ADV-NN` rows;
  T005 does not need to predict T010's id choices beyond using the published
  canonical set.
- **Disposition regex false-positives**: any row cell that happens to
  contain the literal string `OUT-OF-SCOPE` or `DROP` in a non-disposition
  column will pass TC-008 by accident. Mitigation: keep the `justification`
  and `owner_risk` columns free of those exact tokens; prefer lowercase or
  alternate phrasing like "drop candidate" spelled as "removal candidate".
