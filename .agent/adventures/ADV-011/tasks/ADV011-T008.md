Each subsystem gets one paragraph. Every one of the seven keyword tokens required by TC-014 must appear verbatim (case-insensitive) in the paragraph heading:

5. **Evolution** loop — `EvolutionRunner` as a scheduler-driven task; sources: `ark/tools/evolution/evolution_runner.py`, `optimizer.py`, `dataset_builder.py` (ADV-004).

7. **Review** loop — human-in-the-loop gate; source: `ark/tools/visual/review_loop.py` (ADV-006). UI renderers (mermaid, screenshots) are *called by* the review loop, not controllers themselves.

### Section 4 — Event bus, enumerated event kinds (required for Design spec)

One process-wide in-memory bus with an append-only jsonl mirror at `.agent/events.log` (per pattern "Append-only jsonl for multi-writer state" from ADV-007). Exactly these event kinds — list them in this section as an explicit enumerated list:

1. `task.created`
2. `task.completed`
3. `task.failed`
4. `artefact.written`
5. `subagent.started`
6. `subagent.stopped`
7. `skill.triggered`
8. `skill.generated`
9. `skill.improved`
10. `evolution.started`
11. `evolution.converged`
12. `review.requested`
13. `review.completed`
14. `scheduler.tick`

Every event carries the common envelope `{event_kind, adventure_id, task_id?, agent, ts, session_id?, payload}`. Subscribers: telemetry capturer, scheduler, skill-manager improvement hook, review loop. The event bus is the single integration point — no subsystem may read another subsystem's internal state to learn about things; it must subscribe to an event kind.

### Section 5 — State model (three lanes)

Three, and only three, persistent state surfaces. List them in a table:

| Lane | File / location | Shape | Writer | Readers |
|---|---|---|---|---|
| Persistent memory | `ark/tools/controller/memory/*.json` (from ADV-005 `learning.py`) | session entries, generated skills | skill-manager, learning split | controller spine |
| Metrics rows | `.agent/adventures/{ADV-ID}/metrics.md` (ADV-010 row schema) | pipe-table append-only | telemetry subsystem | aggregator, cost review, backfill |
| Adventure log | `.agent/adventures/{ADV-ID}/adventure.log` | one line per step, append-only | every agent role | humans, backfill |

The controller commits to append-only jsonl for event mirror; rendered markdown views (metrics.md, adventure.log) are regenerable from the event log. No in-place rewrites, no shared mutable state.

### Section 6 — ADV-010 telemetry merge mechanics (required for TC-013)

This section is the hard requirement: telemetry is *inside* the controller, not a sibling. Spell out:

1. **Hook ingress stays declarative.** The `SubagentStop` and `PostToolUse` hooks defined in `.claude/settings.local.json` (per ADV-010 `design-hook-integration.md`) remain the trigger; their command stays `python .agent/telemetry/capture.py` for back-compat, but that entrypoint is a thin wrapper that now publishes `subagent.stopped` (or `artefact.written` for PostToolUse) on the controller's event bus and returns — rather than directly writing `metrics.md`.
2. **Capture becomes a subscriber.** The row-writer logic from ADV-010 `design-capture-contract.md` (SubagentEvent → MetricsRow → cost lookup → append row) becomes a subscriber to `subagent.stopped`. Idempotency via Run ID (`sha1(adventure_id|agent|task|model|timestamp|session_id)`) is preserved at the subscriber, not the hook.
3. **Cost model and aggregator are controller services.** `cost_model.cost_for(...)` and the aggregator pass are plain functions called by the subscriber. They are not parallel processes.
4. **Backfill stays offline.** `design-backfill-strategy.md`'s `backfill.py` tool remains a human-invoked CLI that reads `adventure.log`, git history, and existing rows. It does not join the event bus; it writes to `metrics.md.new` with the `Confidence` column set to `medium`/`low`/`estimated`. Hook-written rows keep `Confidence: high`. The report must cite `design-backfill-strategy.md` explicitly when describing backfill.
5. **Error isolation preserved.** The `capture.py` wrapper keeps the exit-0-on-failure guarantee from ADV-010 `design-error-isolation.md`, so a controller-internal failure never propagates back into the Claude Code hook.
6. **Schema imports.** The SubagentEvent dataclass from ADV-010 `schemas/event_payload.md` is the canonical shape for the `subagent.stopped` event payload inside the controller. No second shape.

### Section 7 — Reflexive-dogfooding validation practice

One paragraph plus a three-item checklist. Pattern source: reflexive dogfooding as captured across ADV-002 / ADV-004 / ADV-005 / ADV-008. The practice statement:

> The controller's first consumer is Ark itself. Every subsystem is validated by running it over an in-repo artefact before any external consumer is considered: Scheduler drives `.agent/adventures/*` task progression; EvolutionRunner evolves `.agent/roles/*.md` prompts; Evaluator scores generated Ark specs; Telemetry captures this very adventure's runs into `.agent/adventures/ADV-011/metrics.md`; Review loop gates the unified-controller design itself. This is a *validation practice*, not a subsystem — it is how we know the design survives contact with a real consumer before we cut a downstream adventure.

Checklist (every subsystem must satisfy at least one):
- [ ] Subsystem has been exercised over an in-repo artefact during this adventure.
- [ ] That exercise produced an observable side-effect (a row, a log line, a file).
- [ ] The side-effect is the same shape it will produce for an external consumer.
