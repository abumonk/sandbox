## ADV-DU Descriptor Unification

### Concept

ADV-DU realises the unified descriptor design produced by ADV-011 T006. It
reshuffles the nine `ark/dsl/stdlib/*.ark` files from the current flat layout
into the target two-level domain layout (`stdlib/primitives/` and
`stdlib/<domain>/`), ports both the Lark (Python) and Pest (Rust) grammars to
the full unified `Item` enum covering all six item kinds, and closes every
`DEFERRED-TO: ADV-DU` verdict in validation-coverage.md (31 rows across
ADV-001..ADV-008). This adventure touches only the descriptor layer — grammar
files, stdlib artefacts, the Python `ArkFile` parser class, and the Rust
`ark-dsl` crate's AST — and introduces no new builder or controller
capabilities.

### Scope

- `ark/dsl/stdlib/` — restructure into `stdlib/primitives/` and
  `stdlib/<domain>/` subdirectories per descriptor-delta.md verdict table.
- `ark/dsl/grammar/ark.lark` — extend Lark grammar to accept all six `Item`
  kinds and the full param-ref sigil set.
- `ark-dsl/src/grammar/ark.pest` — mirror Lark additions in the Pest grammar.
- `ark/dsl/parser.py` (`ArkFile` class) — populate `roles`, `studios`,
  `commands`, `hooks`, `rules`, `templates`, `expressions`, `predicates`
  indices; emit `ArkParseError` with `file:line:col` for malformed items.
- `ark-dsl/src/ast.rs` — extend `Item` enum with all item-kind variants
  confirmed by descriptor-delta.md#ast-family-spec.
- `docs/DSL_SPEC.md` — document expression/predicate and the two-level stdlib
  layout (closes TC-029 from ADV-001).
- Backlog entries and `root.ark` `SystemRegistry` registrations for all island
  specs deferred from ADV-001..ADV-008.
- No new capabilities beyond closing the 31 DEFERRED-TO: ADV-DU verdicts.

### Depends on

none

### Est. task count

12–16 — one task per stdlib file move/rename (9 files), plus grammar authoring
(Lark + Pest in parallel), parser index extension, AST enum update, and docs
close-out. Stays within the ≤20 tasks/adventure envelope.

## ADV-BC Builder Consolidation

### Concept

ADV-BC realises the unified builder design produced by ADV-011 T007. It
implements the four canonical verify passes (`numeric_interval`,
`opaque_primitive`, `reference_exists`, `dag_acyclicity`) as a shared library
consumed by all domain builder modules, wires the domain-specific residue
passes (studio rule satisfiability, tool-permission cross-check, evolution
invariants), and completes the codegen harness so that every `Item` variant
emits compilable artefacts for the Rust and Python targets. All
`COVERED-BY: builder-delta.md` cells in validation-coverage.md are already
confirmed; this adventure produces the running code that makes them green under
the test harness. It does not touch the descriptor layer or the controller
runtime.

### Scope

- `ark/builder/` (Python) — implement four shared verify-pass functions as
  a `verify_passes` module consumed by all domain builders.
- `ark-core/src/builder/` (Rust) — mirror the four verify passes in Rust;
  expose as `ark_core::builder::passes`.
- Domain builder modules: `ark/builder/studio_builder.py`,
  `code_graph_builder.py`, `evolution_builder.py`, `agent_builder.py`,
  `visual_builder.py` — wire domain-specific residue passes.
- `ark/codegen/` — extend the codegen harness to handle all six `Item`
  variants per builder-delta.md#codegen-emitter-catalog.
- `tests/builder/` and `tests/codegen/` — pytest suites achieving ≥80%
  line coverage on each builder module.
- No new DSL item kinds or grammar changes (those belong to ADV-DU).

### Depends on

ADV-DU

### Est. task count

10–14 — four shared verify-pass tasks (one each), five domain-builder wiring
tasks, two codegen-harness tasks (Rust + Python targets), one test-suite
integration task. Within ≤20 envelope.

## ADV-CC Controller Consolidation

### Concept

ADV-CC realises the unified controller design produced by ADV-011 T008. It
refactors the existing controller surface into seven named subsystems (Session,
Plan, Execute, Verify, Codegen, Observe, Evolve), implements the event bus that
decouples subsystem interactions, defines the three state lanes (task, session,
adventure), and merges the ADV-010 telemetry work into the Observe subsystem.
The Evolve subsystem hosts the phase-4 forward note produced by the unified
controller design but does not implement the full Darwinian evolver loop (that
belongs to ADV-CE). ADV-CC does not touch the descriptor or builder layers.

### Scope

- `ark/controller/` (Python) — decompose into seven subsystem modules:
  `session.py`, `plan.py`, `execute.py`, `verify.py`, `codegen.py`,
  `observe.py`, `evolve.py`.
- `ark-orchestrator/src/` (Rust) — implement the event bus and state-lane
  structs per controller-delta.md.
- `ark/controller/event_bus.py` — Python event-bus shim for subsystem
  decoupling.
- ADV-010 telemetry sink integration into `observe.py` / Observe subsystem.
- `Evolve` subsystem stub with `NotImplementedError` and a phase-4 forward
  note (placeholder for ADV-CE).
- `tests/controller/` — integration tests for the event-bus routing and
  state-lane transitions.
- No new DSL item kinds, no new verify passes (those belong to ADV-DU / ADV-BC).

### Depends on

ADV-DU, ADV-BC

### Est. task count

12–16 — seven subsystem extraction tasks, event-bus implementation, state-lane
struct definition, ADV-010 telemetry merge, and controller integration tests.
Within ≤20 envelope.

## ADV-OP Out-of-Scope Parking

### Concept

ADV-OP executes the pruning catalog produced by ADV-011 T005. It freezes every
`OUT-OF-SCOPE` artefact by adding provenance headers, archiving them to a
designated `archive/` path, and updating any call sites that reference them
with tombstone comments pointing to the downstream adventure that recovers the
artefact (ADV-UI for UI surfaces, ADV-CE for the evolver stub, ADV-DU for the
shape_grammar sibling-package reference). DROP artefacts receive a deletion
commit with a rationale comment in the git message. This adventure does not
implement any new logic; it is a structural hygiene pass that ensures ADV-DU,
ADV-BC, and ADV-CC inherit a clean working set.

### Scope

- `OUT-OF-SCOPE → ADV-UI` rows: `screenshot_manager`, `visual_search`,
  `html_previewer` (ADV-006), `visual screenshot surfaces` (ADV-006),
  `phase4-ui-requirements`, `phase4-ui-architecture` (ADV-007) — archive to
  `archive/ui/` with provenance headers.
- `OUT-OF-SCOPE → ADV-CE` row: `Darwinian git-organism evolver` (ADV-004) —
  archive stub to `archive/evolution/` with ADV-CE forward reference.
- `OUT-OF-SCOPE → ADV-DU` row: `shape_grammar/` package contents (ADV-008) —
  add reference-integration citation in unified descriptor design; no file
  moves required (package stays on disk).
- All `DROP` rows from pruning-catalog.md — deletion commits with rationale
  messages for phase-N ecosystem research artefacts (ADV-007 phases 1–7),
  MCP catalog entries, and speculative concept artefacts.
- `CHANGELOG.md` / adventure backlog — record all parking decisions with
  downstream adventure anchors.
- No new capabilities; no code logic changes.

### Depends on

ADV-DU, ADV-BC, ADV-CC

### Est. task count

6–8 — one task per pruning-catalog disposition group (ADV-UI parking, ADV-CE
parking, ADV-DU reference update, DROP batch for ADV-007 ecosystem artefacts,
DROP batch for MCP/speculation rows, backlog update). Within ≤20 envelope.

- No new descriptor item kinds or verify passes.

pruning catalog and in validation-coverage.md. It implements the six
`DEFERRED-TO: ADV-UI` test conditions (ADV-002 TC-022/TC-023, ADV-004 TC-038,
ADV-005 TC-036, ADV-006 TC-023/TC-031) and unarchives the four

visual_search, html_previewer from ADV-006; visual screenshot surfaces;
phase4-ui-requirements; phase4-ui-architecture from ADV-007). ADV-UI is a
sibling package; it does not depend on the core descriptor/builder/controller
chain, but it is scheduled after ADV-OP so the archive layout is stable before
UI work begins.

### Scope

  advanced modes.

  TC-022: HTML contains code-graph nodes with correct styling).

  TC-023).

  TC-036).
- Visual runner → html_previewer dispatch (closes ADV-006 TC-023, TC-031).

  unpack and use as design inputs.
- No changes to the core descriptor, builder, or controller layers.

### Depends on

none

### Notes

ADV-UI is a sibling package with no compile-time dependency on ADV-DU, ADV-BC,
or ADV-CC. It is scheduled after ADV-OP per the ordering constraint so that the
archived UI artefacts are available as unpacked source inputs when ADV-UI
begins. The six DEFERRED-TO verdicts it closes are UI-rendering concerns that
were explicitly separated from core pipeline work during T009 validation.
