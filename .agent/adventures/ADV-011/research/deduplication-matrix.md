## Not Duplicates

The following multi-source or cross-adventure concept pairs were examined and rejected as true duplicates. Each is legitimately distinct and is intentionally kept as separate concepts in `concept-mapping.md`.

- **Item::Hook (ADV-003) vs SubagentStop hook (ADV-010)** — sources: (ADV-003: Item::Hook DSL item kind); (ADV-010: SubagentStop_hook runtime event). Rationale: `Item::Hook` is a user-authored DSL construct declaring a hook in a studio spec (descriptor), while `SubagentStop` is a runtime event from the Claude SDK that triggers telemetry capture (controller) — they operate at entirely different abstraction levels and have no shared implementation.

- **Item::StudioJob role (ADV-003) vs Item::Agent (ADV-005)** — sources: (ADV-003: Item::StudioJob role — studio agent persona); (ADV-005: Item::Agent — hermes agent DSL item). Rationale: both describe "an agent" but in distinct domain vocabularies with different field schemas and different runtime interpretations — `StudioJob` is a studio-tier role with escalation paths and skill assignments, while `Item::Agent` is a full hermes agent with platform routing, gateway, and learning config; collapsing them would lose domain-specific semantics.

- **review_loop (ADV-006) vs evolution_runner (ADV-004)** — sources: (ADV-006: review_loop orchestrating visual review cycles); (ADV-004: evolution_runner orchestrating the full optimization loop). Rationale: both are runtime orchestrators (controller bucket) but operate over entirely different domains and data — `review_loop` drives visual feedback cycles over diagrams, while `evolution_runner` drives GEPA-style mutation/fitness loops over LLM variants; the shared lifecycle shape is captured in the "Runtime orchestrator pattern" matrix row above, but the two runners remain distinct entities.

- **opaque-usage tracking (ADV-001) vs PASS_OPAQUE outcome** — sources: (ADV-001: opaque_usage_tracking module-global accumulator); (ADV-001/002/003/004/006/008: PASS_OPAQUE Z3 pass result). Rationale: `opaque_usage_tracking` is the module-level accumulator that records which opaque primitive IDs were encountered during a verification run (a side-channel for reporting), while `PASS_OPAQUE` is the Z3 outcome code returned when a pipe stage is undecidable — they are distinct concerns already absorbed into seed row 4 for the registrar unification; the tracking accumulator remains a separate builder concern.

- **semantic label propagation (ADV-008) vs expression_primitives (ADV-001)** — sources: (ADV-008: semantic label propagation in shape grammar derivations); (ADV-001: expression_primitives registry table). Rationale: both are registry/propagation concepts but in entirely different domains — shape grammar terminal label inheritance vs. pipe-stage primitive dispatch — with no semantic overlap; each is a single-adventure concept with no cross-adventure duplication.
