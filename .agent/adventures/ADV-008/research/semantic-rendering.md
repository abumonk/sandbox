# Semantic Rendering via Annotated Shape Grammar Rule Trees

*Produced for ADV-008-T16 by shape-grammar-researcher, 2026-04-14*

---

## Abstract

A shape grammar's derivation tree is more than a recipe for geometry — it is a semantic
record. Every node in the tree carries the rule that produced it, the scope it inhabited,
and the label its author assigned. When evaluation terminates, each terminal shape is not
merely a mesh; it is a provenance-annotated entity whose meaning is legible to downstream
systems. This document argues that this annotated rule tree is the natural "source of truth"
for semantic rendering: material and shader selection, level-of-detail pruning, neural
renderer conditioning, and differentiable visualization can all be driven by the label
hierarchy rather than by ad hoc mesh tags applied after the fact.

**Why this matters for ADV-008.** The `shape_grammar` package produces an OBJ file where
each face group corresponds to a semantic label (`window`, `wall`, `load-bearing`, `glass`,
`abstraction`, `bridge`, etc.). A downstream renderer — whether a classic rasterizer, a
physically based shader graph, or a neural radiance field — can consume those labels
directly. The provenance chain (`rule_chain` on each `Provenance` record) further allows
a renderer to ask "which rule produced this pixel?", enabling traceable procedural content
generation with pixel-level accountability.

---

## Thesis

**The annotated shape-grammar rule tree is a semantic scene description.** Labels
propagate from parent rules to child shapes; provenance chains record the exact derivation
path. When the evaluator emits an OBJ file with per-label face groups, the result is not
just geometry — it is geometry annotated with *why each face exists*. Renderers that
consume this representation gain:

1. **Material selection by class** — no manual UV painting; the label drives the shader.
2. **LOD by semantic pruning** — drop all `decorative`-labeled subtrees at export time to
   produce a lower-detail variant without re-running the grammar.
3. **Neural renderer conditioning** — semantic segmentation maps derived from labels
   provide strong priors for NeRF-style or Gaussian-splat renderers.
4. **Differentiable visualization** — the rule tree can be differentiated with respect to
   grammar parameters; gradient signals flow back through the label assignments.

## Background: Semantic-Rendering Landscape

This section briefly surveys existing approaches to semantic-aware rendering to situate the
shape-grammar approach within the literature.

### Classical Forward Renderers with Material IDs

The oldest approach assigns integer material IDs to mesh faces at asset-creation time. A
material library (OBJ/MTL, glTF materials, Unreal material instances) maps each ID to a
shader. The limitation is that the assignment is static and opaque: a face knows it is
"material 3" but not that it is a "load-bearing wall generated by rule Floor.ground". The
shape-grammar approach replaces the opaque ID with a symbolic label that carries derivation
context.

CityEngine / CGA (Müller et al., "Procedural Modeling of Buildings," SIGGRAPH 2006) already
uses `set("material", name)` calls in grammar rules to propagate material names through
derivations. ADV-008 extends this pattern by coupling the material name to a richer
`SemanticLabel` with a tag list and an inheritance flag, and by adding a `Provenance` record
that captures the full rule chain — information CityEngine discards after evaluation.

### Semantic Segmentation Driven Renderers

Semantic segmentation networks (FCN, DeepLab, Mask R-CNN) produce per-pixel class labels
from images. Several rendering pipelines have used these labels as conditioning signals:
- **SemanticGAN** (Li et al., "Semantic Image Synthesis with Spatially-Adaptive
  Normalization," CVPR 2019 — SPADE) demonstrated that semantic segmentation maps, when
  used as spatial conditioning signals for a GAN generator, produce photorealistic images
  with controllable semantic content.
- **GauGAN / SPADE** showed that a simple 2D semantic map is sufficient to condition a
  full neural renderer.

The shape-grammar approach generates the semantic map procedurally from the rule tree,
rather than labeling a photograph. This provides ground-truth semantic annotations for
free, at any resolution, for any generated scene variant.

### Neural Radiance Fields with Semantic Priors

Neural radiance fields (NeRF, Mildenhall et al., "NeRF: Representing Scenes as Neural
Radiance Fields for View Synthesis," ECCV 2020) learn a volumetric scene representation
from posed images. Extensions such as Semantic-NeRF (Zhi et al., "In-Place Scene
Labelling and Understanding with Implicit Scene Representation," ICCV 2021) add a semantic
prediction head to the NeRF MLP, jointly trained on color and semantic label supervision.

The shape-grammar OBJ with per-label groups can be rendered to produce multi-view images
with per-pixel semantic labels — an ideal training corpus for Semantic-NeRF. The provenance
chain provides an additional supervision signal: a "rule-ID" prediction head could be
trained to predict which rule produced each point in 3D space.

### Signed Distance Fields with Semantic Labels

SDF-based representations (DeepSDF, Park et al., "DeepSDF: Learning Continuous Signed
Distance Functions for Shape Reconstruction," CVPR 2019) encode shape as a learned
continuous function. Semantic extensions annotate the SDF with a per-point label field.
The shape grammar's axis-aligned scope tree maps naturally to an SDF: each scope is an
oriented bounding box whose SDF is analytically known (a box SDF), and the label is
inherited from the rule that produced it. Future work could export the rule tree directly
as a hierarchical SDF with embedded semantic labels rather than as a polygon mesh.

## Shape Grammar → Semantic Rendering

This section explains the concrete pipeline by which the ADV-008 `shape_grammar` package
feeds a renderer.

### Label Inheritance

Every rule in a `ShapeGrammar` island may carry a `SemanticLabel` on its output shapes.
The `semantic.py` propagation pass walks the derivation tree breadth-first:

```
parent shape (label: "wall")
    │
    ├── child shape (no explicit label) → inherits "wall"
    └── child shape (explicit label: "window") → overrides parent label
```

The `SemanticLabel.inherits` flag controls opt-out: a rule can mark `inherits: false` to
prevent its label from propagating further down the tree.

### Provenance Chain

Every terminal shape carries a `Provenance` record with two fields:
- `rule_chain: [String]` — the ordered list of rule names that produced this terminal,
  from the axiom rule down to the immediate parent.
- `depth: Int` — the depth in the derivation tree (0 = axiom, N = terminal).

At OBJ export time, the `obj_writer.py` module groups faces by their `SemanticLabel.name`
and records the `rule_chain` as an OBJ comment on the group header:

```obj
g window
# provenance: Building > Floor > FacadePanel > Window
v 0.0 0.0 0.0
...
```

This makes provenance readable by any OBJ parser that consumes comments.

### Scope Attributes

Each shape's `Scope` carries an attribute bag (`attrs: [AttrEntry]`) whose entries are
inherited from parent scopes. The evaluator merges the parent attribute bag into each child
before firing the child's rule. Attributes set by `set(key, value)` calls shadow parent
entries with the same key.

Scope attributes are the carrier for semantic metadata beyond the primary label: LOD hints,
material variant names, physics-material identifiers, and user-defined tags all travel in
the attribute bag and emerge at terminal shapes, ready for downstream consumption.

### Renderer Integration

A downstream renderer consumes the labeled OBJ via three integration points:

1. **Per-group material binding** — the renderer reads each OBJ `g <label>` group and
   looks up `<label>` in a label-to-material map. No UV mapping required.
2. **Semantic segmentation map generation** — render each group with a distinct flat color;
   the result is a ground-truth semantic segmentation image at any resolution.
3. **Neural renderer conditioning** — pass the segmentation image and the label-to-class
   integer map to a SPADE or Semantic-NeRF model as conditioning input.

### Prototype 1: Semantic Facade → Material-ID Rendering

**Input grammar**: `shape_grammar/examples/semantic_facade.ark`

This grammar describes a procedurally generated building facade with five semantic classes:
`floor`, `window`, `frame`, `glass`, and `wall` (the last sub-labelled `load-bearing` on
the ground floor). The derivation proceeds as follows:

1. **Axiom**: `Building` with a scope of 20 × 40 × 30 (width × height × depth, in metres).
2. **Rule `Building → floors`**: split along the Y axis into N floors, each assigned the
   label `floor`.
3. **Rule `Floor → panels`**: split along the X axis alternating between `window` and `wall`
   panels. Ground-floor walls receive the additional tag `load-bearing`.
4. **Rule `Window → frame + glass`**: split the window scope into an outer `frame` and an
   inner `glass` panel.
5. **Rule `Wall → extrude`**: extrude the wall scope along Z to give it depth; label
   preserved from the parent floor rule.

The OBJ produced by the evaluator contains five distinct `g` groups, one per semantic
label. The group order in the file reflects the depth-first derivation order.

**Expected output file structure**:

```obj
# shape_grammar evaluator output — seed 42
# grammar: shape_grammar/examples/semantic_facade.ark
g floor
# provenance: Building > Floor
v ...
f ...
g window
# provenance: Building > Floor > Window
v ...
f ...
g frame
# provenance: Building > Floor > Window > Frame
v ...
f ...
g glass
# provenance: Building > Floor > Window > Glass
v ...
f ...
g wall
# provenance: Building > Floor > Wall
v ...
f ...
```

**Label-to-material map** (used by the Blender / Three.js / neural renderer adapter):

| Semantic label | Material / color |
|----------------|-----------------|
| `floor`        | concrete grey (`#808080`) |
| `window`       | placeholder (replaced by `frame` + `glass`) |
| `frame`        | brown wood (`#8B4513`) |
| `glass`        | cyan translucent (`#00FFFF`, alpha 0.3) |
| `wall`         | light grey (`#C0C0C0`) |
| `load-bearing` (wall tag) | dark grey (`#404040`) |

A Blender Python script reads the OBJ groups and assigns material nodes by iterating the
label map. Three.js achieves the same effect by setting `MeshStandardMaterial` instances
per group on import. A neural renderer (e.g., SPADE/GauGAN) uses the flat-shaded render of
this labeled scene as its semantic segmentation conditioning map.

**Provenance consumption by a neural renderer**: the `rule_chain` embedded in OBJ comments
can be parsed to construct a one-hot "rule-ID" feature per face group. This feature,
concatenated with the semantic class index, forms a richer conditioning signal than a raw
class label alone — it encodes not only *what* the face is, but *how* the grammar produced it.

**Proof command**:

```bash
python -m shape_grammar.tools.evaluator shape_grammar/examples/semantic_facade.ark --seed 42 --out /tmp/facade.obj
```

Running this command should exit 0 and write `/tmp/facade.obj` containing at minimum the
five `g` group directives listed above, with OBJ comment provenance headers on each group.

### Prototype 2: code_graph_viz.ark → Ark-Spec Visualization

**Input grammar**: `shape_grammar/examples/code_graph_viz.ark`

This grammar closes the loop from Ark specs back to rendered geometry. It reads the IR of
`ark/specs/infra/code_graph.ark` (the island that defines the `code_graph` data structure
from ADV-007) and converts each structural element into a procedurally placed 3D primitive
with a semantic label matching its Ark kind:

1. **Input phase**: the evaluator reads `code_graph.ark` via `ark parse` (subprocess) to
   obtain its AST. The extracted entity list is passed to the grammar as external parameters
   (a seed-stable JSON sidecar).
2. **`abstraction` entities** → spheres of radius 1.0, placed on the outer ring of a
   radial layout. Label: `abstraction`. Color: indigo (`#4B0082`).
3. **`class` entities** → axis-aligned cubes whose side length scales with the entity's
   `$data` field count (minimum 0.5, maximum 3.0). Label: `class`. Color: teal (`#008080`).
   Classes are placed inside the outer ring, evenly distributed.
4. **`bridge` connections** → cylinders of radius 0.1 connecting the center of each pair of
   linked shapes. Label: `bridge`. Color: orange (`#FFA500`).
5. **Layout algorithm**: a radial plan. Abstractions occupy angles `[0°, 360° / n_abstractions)`;
   classes fill the interior on a sub-ring at 60 % of the outer radius; bridges are drawn
   as straight cylinders between connected shape centers.

The derivation tree mirrors the Ark island's structural hierarchy: `abstraction` nodes are
parents of the `class` nodes that implement them; `bridge` nodes are children of the
pair of shapes they connect. Semantic labels propagate automatically — `abstraction` shapes'
children (class nodes) inherit the `abstraction` label unless they override it with `class`,
which they do explicitly.

**Expected output file structure**:

```obj
# shape_grammar evaluator output — seed 7
# grammar: shape_grammar/examples/code_graph_viz.ark
g abstraction
# provenance: CodeGraphViz > OuterRing > AbstractionSphere
v ...
f ...
g class
# provenance: CodeGraphViz > InnerRing > ClassCube
v ...
f ...
g bridge
# provenance: CodeGraphViz > BridgeCylinder
v ...
f ...
```

**Closing the loop — dogfood semantics**: this prototype demonstrates that the same
toolchain that generates building facades can generate a 3D visualization of the Ark spec
system itself:

```
Ark island (code_graph.ark)
    │  [ark parse]
    ▼
AST JSON
    │  [code_graph_viz.ark grammar + shape_grammar evaluator]
    ▼
Labeled OBJ (abstraction / class / bridge groups)
    │  [Blender / Three.js importer]
    ▼
Interactive 3D graph of the Ark spec structure
```

The OBJ can be re-imported by the `ark visualize` adapter (ADV-007 infrastructure) to
produce an HTML-based interactive view. This creates a feedback cycle: each change to a
`.ark` spec file produces a new procedurally generated visualization without any manual
modeling. The semantic labels on the OBJ groups drive the Three.js material assignments
automatically, so the visualization stays consistent as the spec evolves.

**Proof command**:

```bash
python -m shape_grammar.tools.evaluator shape_grammar/examples/code_graph_viz.ark --seed 7 --out /tmp/code_graph.obj
```

Running this command should exit 0 and write `/tmp/code_graph.obj` containing the three
`g` group directives (`abstraction`, `class`, `bridge`) with provenance comment headers.
The number of faces in each group varies with the content of `code_graph.ark` at evaluation
time.

## Why Rule Trees Are a Natural Semantic Source

Three properties of the shape-grammar rule tree make it a superior semantic source compared
to post-hoc mesh annotation:

1. **Labels are structural, not incidental.** A `window` label on a terminal shape reflects
   that the grammar author explicitly wrote `Window → frame + glass` with a label override.
   The label cannot be wrong without the grammar being wrong. Post-hoc annotation can
   disagree with the mesh topology; rule-tree labels cannot.

2. **Provenance is free.** Every derivation step is already recorded; the `rule_chain`
   list costs one string append per rule application. There is no retroactive reconstruction
   problem.

3. **Labels compose with stochastic evaluation.** When `--seed 42` produces a facade with 8
   floors and `--seed 99` produces one with 12, the labels are structurally consistent
   across both outputs. A renderer consuming the semantic groups needs no retraining for
   the new geometry — the label contract is grammar-level, not geometry-level.

## Integration Points with Ark

The `shape_grammar` package integrates with the Ark toolchain at three points relevant to
semantic rendering:

1. **`ark parse`** — used by `code_graph_viz.ark` to extract the AST of a target `.ark`
   file for visualization input. The parsed AST provides entity names, field counts, and
   bridge declarations without any Ark modification.

2. **`ark visualize` adapter** — the ADV-007 `ark graph` command writes an HTML file
   visualizing an island's structure. The semantic OBJ from Prototype 2 is a geometry-level
   complement to this 2D graph view; together they provide both a schematic and a 3D
   spatial rendering of the same spec.

3. **`shape_grammar/tools/semantic.py`** — the `propagate()` function runs after evaluation
   and before OBJ export. It ensures every terminal has a non-null `SemanticLabel` (falling
   back to the parent's label if the child did not override). The labeled terminal set is
   then consumed by `obj_writer.py` for group-per-label OBJ serialization.

## Further Work

- **Neural renderer training on the provenance chain.** The `rule_chain` field on each
  terminal provides a sequence-structured supervision signal beyond a flat class label.
  A sequence-to-sequence model (e.g., a Transformer with positional encoding over rule
  depth) could be trained to predict the rule chain from rendered pixels, enabling
  inverse procedural modeling: given a photograph of a building, recover the grammar rules
  that likely produced it.

- **GPU evaluator for high-resolution grammars.** The current Python evaluator is
  single-threaded and CPU-bound. A GPU-parallel evaluator (CUDA kernel or WebGPU compute
  shader) could evaluate thousands of stochastic variants in parallel, each with a
  different seed, to generate large training datasets for semantic renderers without
  bottlenecking on serial rule application.

- **Live-editing feedback loop.** Connect a grammar editor (VS Code language server using
  `shape_grammar/tools/evaluator.py` as a language server backend) to a Three.js viewport.
  Every grammar file save triggers a re-evaluation and live-updates the semantic OBJ in the
  viewport. The semantic label structure ensures that material assignments persist across
  edits that do not change the label names — only geometry changes.

- **LOD via rule-tree pruning.** Implement an evaluator flag `--drop-labels load-bearing
  decorative` that terminates derivation early for shapes carrying the listed labels,
  emitting a simplified placeholder mesh instead. This produces LOD variants as a
  post-processing step on the rule tree, without authoring a separate lower-detail grammar.

- **Dogfood: adventure and workflow graph visualization.** Extend Prototype 2 from
  `code_graph.ark` to the full adventure manifest (`ADV-008/manifest.md`). Each task
  becomes a `class` node, each dependency a `bridge`, and each phase cluster an
  `abstraction`. The resulting 3D graph provides a spatial overview of the adventure's
  dependency structure, rendered by the same semantic-rendering pipeline.
