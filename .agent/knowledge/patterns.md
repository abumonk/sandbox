# Patterns

Recurring patterns discovered across tasks.

- **Dual-grammar parity**: When extending Ark's grammar, implement pest (Rust) and Lark (Python) rules in parallel with matching rule names. T002+T005 demonstrated this works cleanly when design docs specify both grammars upfront. (from ADV-001)
- **Test strategy before implementation**: Creating a test strategy document (T001) that maps all target conditions to specific test functions and files before any implementation begins provides a clear contract for implementers and reviewers. Reduces ambiguity about what "done" means. (from ADV-001)
- **AST-over-tree-sitter**: For Python parsing, use the built-in `ast` module instead of tree-sitter to avoid native binary dependencies. Supplement with regex for languages without built-in AST support (e.g., Rust). (from ADV-002)
- **Reflexive self-indexing**: When building analysis tools, run them on their own codebase as the first real consumer. This validates integration end-to-end and catches issues before formal testing. ADV-002 indexed 771 nodes / 3475 edges from Ark's own sources. (from ADV-002)
- **Separate Domain Modules**: When adding a new DSL domain (e.g., studio), create dedicated `{domain}_verify.py` and `{domain}_codegen.py` modules rather than extending core files. Keeps concerns isolated and core pipeline stable. (from ADV-003)
- **Design-First Zero-Rework**: Creating detailed design docs with explicit acceptance criteria and proof commands enables all tasks to pass review on first submission with zero rework iterations. Validated across 3 adventures: ADV-003 (14 tasks, 7 designs), ADV-004 (19 tasks, 10 designs, 46 target conditions), ADV-005 (21 tasks, 11 designs, 44 TCs) — 54 total tasks, 0 rework. Invest time in design upfront. (from ADV-003, ADV-004, ADV-005)
- **Evolution Module Architecture**: Large subsystem features (7+ item types) benefit from a dedicated `tools/{domain}/` package with separate modules for each concern (dataset_builder, fitness, optimizer, constraint_checker, runner) plus dedicated verify, codegen, and visualizer extensions. This keeps the package self-contained while integrating cleanly with the existing pipeline. ADV-004 delivered 5 evaluation modules + 3 integration modules with zero coupling issues. (from ADV-004)
- **Reflexive Dogfooding**: When building a DSL subsystem, create specs that use the subsystem to describe itself (e.g., evolution_skills.ark uses evolution items to define how to optimize Ark skills). This validates end-to-end composability and serves as living documentation. ADV-004's reflexive specs caught 2 bugs in evolution_verify.py during T016 implementation. (from ADV-004)
- **Layered Agent Runtime**: When building a multi-subsystem runtime (gateway, backends, skills, learning, scheduler), use a top-level orchestrator (AgentRunner) with dependency injection for all subsystems. This enables independent testing of each module and flexible composition. Build modules bottom-up (types -> individual modules -> orchestrator -> integration) with tests at each layer. (from ADV-005)
- **Dual-input AST helpers**: When codegen/verify modules consume AST data that may arrive as either dict (JSON) or dataclass (Python), use `_get(obj, field)` helpers that dispatch on type. This makes modules resilient to AST shape changes and enables both CLI (JSON) and programmatic (dataclass) usage paths. (from ADV-005)
- **Visual Subsystem as Island**: When adding a major feature domain (visual communication, studio, etc.), structure it as a four-layer island: DSL types + grammar rules -> tool modules -> pipeline integration (CLI, verify, codegen) -> self-referential specs. ADV-006 delivered 19 tasks with zero rework using this pattern. (from ADV-006)
- **Optional Dependency Guard**: For tool modules with heavy dependencies (Pillow, mermaid CLI, etc.), use try/except import guards with a `_AVAILABLE` flag and provide a degraded fallback (e.g., JSON overlay instead of image rendering). Keeps Ark installable without optional deps. (from ADV-006)
- **Check infra specs before recommending external tools**: When researching external tools for adoption, first grep `ark/specs/infra/*.ark` for an existing planned subsystem with the same purpose. If one exists (e.g., `code_graph.ark` already declares `GraphStore/CodeIndexer/QueryEngine/Watcher`), the recommendation shifts from "adopt" to "wrap behind the existing abstraction" — preserves the declarative spec and lets the implementation be swapped later. (from ADV007-T012)
- **Per-role MCP enabling over global enabling**: Every active MCP inflates each role's tool-list prompt. For multi-MCP ecosystems, attach MCP capabilities to specific roles (orchestrator gets github+memory; researcher gets memory+firecrawl; deploy roles get the platform suite) rather than enabling globally. Aligns with token-economy goals and reduces tool-selection ambiguity. (from ADV007-T015)
- **Tier rather than rank external tools**: When cataloging many external tools (>10), classify into discrete tiers (CORE/OPTIONAL/SKIP) rather than ranking 1..N. Tiers force a real adoption decision per tool, while ranks invite adopting "the top N" indiscriminately. (from ADV007-T015)
- **Synthesis matrix from N research docs**: When merging multiple research documents into a decision matrix, follow these rules: (1) phase coverage first — zero-phase tools skip; (2) every cell answers "would the phase fail without this?" — if no, downgrade primary→secondary; (3) surface single-decision blockers as a first-class column, not buried in prose; (4) tag conflicts as redundant (force choice) vs complementary (separate substrates) before recommending; (5) introduce a "native re-implementation" tier for tools whose patterns are valuable but binaries drag toolchain weight; (6) list both critical-path ordering AND priority ordering — they are different. (from ADV007-T016)
- **Carry-forward blockers between research tasks**: Sibling research tasks (T012-T015 here) often surface decisions that block adoption (e.g., "single deploy platform pick blocks 5 MCPs"). The synthesis task (T016) should explicitly enumerate these blockers with owner + trigger, not silently inherit them. Prevents the matrix from looking actionable when it actually has 5+ pending decisions. (from ADV007-T016)
- **Cross-project audit synthesis: severity labels + edge contracts**: When synthesizing N project audits into one cross-project doc, (1) put a single explicit `severity:` token per issue so `grep -c "severity:"` gives a verifiable catalog count; (2) build an edge-by-edge contract table (consumer←producer, transport, payload, contract source, enforcement, status) — this surfaces "half-implemented contracts" and "MCP-optional with try/fallback" anti-patterns that prose-only synthesis hides; (3) add a data-ownership matrix showing which state is written by ≥2 projects — those rows are the unification priorities. (from ADV007-T006)
- **Append-only jsonl for multi-writer state**: For any file-based state that has >1 legitimate writer, promote it from a single mutable markdown file (read-modify-write) to an append-only `.jsonl` with one JSON event per line plus a regenerable rendered view. Each writer appends one sub-PIPE_BUF line (atomic on POSIX and NTFS); readers scan the tail. The rendered view preserves human-readable / grep-based consumers. Resolves race conditions without introducing locks on the hot path. (from ADV007-T008)
- **File→directory promotion for high-traffic entities**: When a single markdown entity (task, lead-state, permissions, manifest) accumulates contention from growing sub-sections (`## Log`, `last_event`, TC rows, eval rows), promote the file to a directory containing (a) a small header file with single-writer fields, (b) an append-only `.jsonl` per growing section, and (c) a `views/` sibling with regenerable rendered markdown. Shrinks per-spawn token context, eliminates RMW races, keeps legacy readers working via the rendered view. (from ADV007-T008)
- **Phased non-breaking migration (M0 shadow → M6 cleanup)**: For entity-schema changes to a live markdown substrate, phase the migration as: M0 add new artefacts alongside old (shadow); M1 regenerate-and-diff until clean; M2 promote new as source of truth with rendered-view shims; M3-M5 incremental surface-by-surface replacement; M6 remove shims + hand-curated catalogs. Each phase is independently shippable and rollback-able; no flag-day migration. (from ADV007-T008)
- **Break mutual-dep cycles by degraded-first landing**: When new subsystems form a strongly-connected cluster (A↔B↔C), do not block on "land all three at once." Land A first in a degraded mode that stubs the B/C dependencies (no retry, no batching, no escalation); land B next with a direct-call path to A and a stubbed C; land C consuming both; then enable the mutual features (retry-backoff, batching, inbox relay). Converts an impossible simultaneous release into 3 shippable ones. (from ADV007-T018)
- **Interaction matrix: new × existing with C/R/U/D/O cells**: When designing N new concepts layered over M existing entities, produce an explicit N×M matrix using Create/Read/Update/Delete/Observe letters per cell. Reveals (1) cross-cutting rows = concepts that touch everything (integration risk), (2) cross-cutting columns = entities that need schema changes for every concept (migration risk), (3) empty quadrants = likely over-scoped concepts. More actionable than prose dependency lists. (from ADV007-T018)
- **MV sketch + growth path per concept**: For every new concept in a roadmap, require a "minimum viable sketch" (smallest shape that proves the idea — often 2-3 tools, 1 UI surface) alongside a "growth path" (what arrives once adoption justifies it). Prevents both scope creep in MV and vision loss in execution. Makes complexity estimation per concept tractable (count the tools + UI surfaces in MV). (from ADV007-T018)
- **Single scalar index for multi-axis reduction work**: When a refactor targets multiple axes (LOC, duplicated schemas, API surface, auto-inject tokens), define one weighted geometric mean (`lightweight_index = prod(L_i^w_i)`) where each `L_i` is the post-reduction fraction on that axis. A single scalar lets every milestone verify "did we shrink the index?" instead of re-running an N-row table per review. Caveat: geometric mean, not arithmetic — a single axis regressing to zero should dominate, not average out. (from ADV007-T020)
- **Per-milestone one-autotest gate rule**: For iterative refactors with many stages, bind each milestone to exactly one named autotest that certifies closure. If a feature needs to be partly done to unblock the next milestone, the milestone splits — never acquires a second gate. Paired with a coverage budget per milestone, this makes progress objectively verifiable by CI rather than review consensus. (from ADV007-T020)
- **Forward-only trapdoors listed explicitly in rollback strategy**: In any multi-stage refactor with feature flags, some decisions (deleting hook prompts, retiring a parser, removing a compat directory) are one-way by design. List those trapdoors explicitly in the strategy doc with the gate-autotest + cycle-count precondition that authorises each one, and require an entry in `adventure.log` citing the authorising gate run when the step is taken. Prevents "oh no, we removed the bash path" surprises. (from ADV007-T020)
- **Abstract representation before concrete stores**: When a domain has multiple concrete storage surfaces (markdown, jsonl, DB, event wire-format), define the abstract algebra first (5 primitives: Atom/Record/Union/List/Stream; 5 ops: read/append/set_header/view/subscribe) and treat every concrete store as a renderer + parser pair over the algebra. Every renderer ships with a round-trip property test; concrete-store divergence becomes impossible by construction. Cheaper than reconciling N hand-written adapters. (from ADV007-T020)
- **Sibling-package DSL consumer**: When a new capability can be built as a *consumer* of an existing DSL (rather than an extension of it), prefer a sibling package with a strict one-way dependency. Keeps the host DSL pristine, avoids core-language churn, and surfaces genuine host-language ergonomics. ADV-008 used this for `shape_grammar/ -> ark/` — 0 Ark modifications, 19 tasks, 27 TCs passed. (from ADV-008)
- **Baseline-diff snapshot**: When a target condition asserts "area X is unmodified" (e.g. `git diff -- X` empty) but X already has pre-existing uncommitted drift from prior work, the TC will fail at wave 0. Mitigation: at adventure start, capture `git diff master -- X > .agent/adventures/{ADV}/baseline-X.diff` and redefine the TC proof as `diff <(git diff master -- X) baseline-X.diff` (must equal empty). Preserves the invariant's intent ("this adventure made no changes to X") without requiring a clean-slate workspace. (from ADV-008 TC-10)
- **Integration-drift catch at downstream review**: When a feature is split across adjacent tasks by file (e.g. producer module in T-n, consumer CLI in T-(n-1)), the wiring can silently go stale — the producer task's reviewer has no reason to check the consumer wiring. The downstream task's reviewer, exercising the full pipeline, catches it. Lesson: reviewers for the *last* task in a multi-task feature must run the end-to-end proof, not just local unit tests. ADV-008 T13 reviewer correctly FAILED when T12's evaluator CLI still called a local stub instead of T13's write_obj; fix was a lazy import to break a circular `Terminal` dep. (from ADV-008)
